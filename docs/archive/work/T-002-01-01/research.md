# T-002-01-01 — Research: ci-module-bootstrap

*Descriptive map of the ground this ticket touches. No solutions here — those are Design.*

## What the ticket asks

Stand up `/ci` as its **own Dagger module** — a separate program with its own
`package.json` on the **Node** runtime, engine pinned to **`v0.21.4`** in
`/ci/dagger.json`, and a **thin** `/ci/src/index.ts` router with **no gate logic**.
This is the shell only. The `test()` gate is the *next* ticket (T-002-01-02); the
`dagger call` no-drift proof is the one after (T-002-01-03).

## The repo as it stands today

- **App is a Bun project at the repo root.** `package.json` (`name: vend`, `type: module`,
  `engines.bun >=1.3.9`) already exposes the command surface that is the *entire*
  contract with `/ci`:
  - `check:test` → `bun test`
  - `check:typecheck` → `tsc --noEmit`
  - `check` → `baml:gen && check:typecheck && check:test`
  - `build` → `tsc --noEmit`
- **`bun run check:test` is green standalone right now:** `229 pass / 0 fail` across
  15 files in ~172ms. (Confirmed on the machine this session.) This matters: the gate
  this story eventually wraps already runs and passes, so the shell is being built on
  solid ground.
- **App `tsconfig.json`** has `"include": ["src"]` — it only compiles the app's `src/`.
  Anything placed at `/ci` is **outside** the app's compile graph by construction.
- **Source layout** (`src/…`): `cli.ts`, `shelf/`, `play/`, `gate/`, `executor/`,
  `log/`, `budget/`, `baml/`, plus `baml_src/` + generated `baml_client/`. None of it
  has any relationship to CI; there is no existing `/ci` directory.
- **No `workspaces` field** in the app `package.json` → a sibling `/ci/package.json`
  will be an independent program, not a workspace member. `bun install` at root will
  not adopt it.

## Tooling on the machine (verified this session)

- **Dagger CLI `v0.21.4`** installed at `/opt/homebrew/bin/dagger`
  (`image://registry.dagger.io/engine:v0.21.4`, darwin/arm64). Exactly the pinned

- **Docker daemon is up.** (Only *needed* for T-002-01-03's `dagger call`; not for

- The CLI nags that `v0.21.7` is available. Per the andon, that upgrade is a **reviewed

## Ground truth: what a v0.21.4 TS module actually looks like

Generated a throwaway module in `/tmp` (`dagger init --sdk=typescript`, default source)
to capture the *authoritative* file shapes for this exact CLI — then deleted it. The
root-source layout (matching `ci-strategy.md`'s diagram) is:

```
/ci
  dagger.json        { name, engineVersion: "v0.21.4", sdk: { source: "typescript" } }
  package.json       { "type": "module", "dependencies": { "typescript": "5.9.3" } }
  tsconfig.json      experimentalDecorators; paths map @dagger.io/dagger -> ./sdk
  src/index.ts       @object() class Ci { @func() ... }
  .gitignore         /sdk, node_modules, .pnpm-store, .env
  .gitattributes     /sdk/** linguist-generated
  sdk/               GENERATED by codegen — gitignored, not committed
  yarn.lock          generated by codegen
```

Load-bearing facts learned:

1. **`dagger init` pins `engineVersion` to the running CLI** (`v0.21.4`) in the file —
   it does *not* write `"latest"`. So a hand-authored file with `v0.21.4` is exactly
   what the tool itself would produce.
2. **`source` is omitted from `dagger.json` when it is the module root** — the diagram's
   "`package.json` at `/ci` root, code in `/ci/src/`" is the *default* layout.
3. **`@dagger.io/dagger` is NOT an npm dependency.** It resolves via a tsconfig `paths`
   mapping to `./sdk/index.ts` — and `sdk/` is **generated** (by `dagger develop` /
   codegen) and **gitignored**. The module's only real npm dep is `typescript` itself.
   The runtime is **Node** (the SDK object/func decorators run under Node in-container),
   never Bun-as-orchestrator.
4. The generated `index.ts` ships two demo functions (`containerEcho`, `grepDir`) — these
   are illustrative gate-ish logic and must be stripped for a "no gate logic" shell.

## The boundary that rots first (the thing to guard hardest)

`ci-strategy.md` (the Central Rule) and `ci-structural-gate.md` (the process) agree:
**Dagger invokes; it never defines.** The *only* contract between `/ci` and the app is
the `bun run check:*` command surface. Therefore:

- `/ci` imports **nothing** from the app's source. The app imports **nothing** from `/ci`.
- `index.ts` is a **thin router** — one gate = one sub-class = one file. TypeScript's
  "can't split the main module file" constraint is the *anti-mess architecture handed to
  us*: it forces sub-class-per-file if adopted deliberately. For this shell there are
  **zero** gates yet, so the router is an empty `@object()` class awaiting `test()`.

## The andon (constraints, stated as stop-conditions)

- **Do not run `dagger develop`** to regen against a newer engine. A version bump is a
  reviewed human step (`ci-strategy.md` rule 1). The pinned `v0.21.4` stays put.
- Because `sdk/` is produced by codegen and codegen is the andon-guarded step, this shell
  is **hand-authored**; `sdk/` is deliberately absent and surfaced as a downstream step.
- Docker daemon must be up for `dagger` to *provision* its engine (~18.4s cold-start,
  measured 2026-06-18) — but provisioning is only exercised by T-002-01-03, not here.

## Assumptions carried into Design

- This ticket's success is *structural*: the shell's files, boundaries, and pinned
  version — **not** a working `dagger call` (that is T-002-01-03).
- Committing increments directly to the shared `main` branch is the established loop
  pattern (lisa concurrency model; every prior ticket did so).
- The app's `check:test` / `check:typecheck` must remain **untouched and green** — the
  shell must not perturb the app's compile or test graph.
