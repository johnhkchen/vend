## 1. `baml_src/decompose.baml`

### 1a. `class StoryDraft` — five optional fields appended after `tickets`

```baml
class StoryDraft {
  id string ...            // existing six fields byte-identical
  ...
  tickets string[] ...
  scope string? @description("the seams/files this story touches and the pipeline slice it owns — where the work lives")
  storyAcceptance string? @description("the observable end-state that proves the story landed — a check or behavior, never a restated title")
  honestBoundary string? @description("what this story deliberately defers and why — free-vs-metered, fixture-vs-live — named, not hidden")
  waveRationale string? @description("why the ticket DAG parallelizes the way it does — which tickets run alone, which fan out, and what makes that safe")
  outOfSlice string? @description("adjacent work this story explicitly does NOT do, so scope creep has a fence to hit")
}
```

Optionality (`?`) is the load-bearing choice (design D1): absence must arrive as `null`, never
drop the story or fabricate a default. Descriptions ride `{{ ctx.output_format }}` to the model.

### 1b. The prompt — one new section between the admit-criteria and "## The epic to clear"

```
## Every story is a CONTRACT (fill all five, honestly)

A story is what a downstream worker builds from without ambient context. Populate ALL five
contract fields on every story:
- `scope` — …            (one demand line per field, naming the JSON field name)
- `storyAcceptance` — …
- `honestBoundary` — …
- `waveRationale` — …
- `outOfSlice` — …

The bar, drawn from a real cleared story:
<exemplar block — byte-identical to STORY_CONTRACT_EXEMPLAR, ~15 lines>

If you cannot fill a section truthfully, leave it ABSENT — a completeness gate refuses shell
stories, and padding a section to pass is worse than absence.
```

Placement rationale: the admit-criteria govern *what work to admit*; this section governs *what a
story artifact must carry* — it reads as the sixth demand, before the per-cast inputs begin.

## 2. `src/play/decompose-epic-core.ts` — the contract's canonical home

Appended at the end of the module (after the graph-integrity section), one new commented section
`── story contract (T-066-01-01) ──`:

```ts
/** The five story-contract field names, exactly as they appear on StoryDraft / in the prompt.
 *  Single source for the render test (each name must be demanded in the prompt) and for
 *  T-066-01-02's completeness gate (the fields it refuses absence of). */
export const STORY_CONTRACT_FIELDS = [
  "scope", "storyAcceptance", "honestBoundary", "waveRationale", "outOfSlice",
] as const;

export type StoryContractField = (typeof STORY_CONTRACT_FIELDS)[number];

/** The in-prompt exemplar of a contract-quality story (condensed from the hand-authored
 *  S-066-01.md). MUST stay byte-identical to the block authored in decompose.baml's prompt —
 *  the render test asserts containment, so drift stops the line. */
export const STORY_CONTRACT_EXEMPLAR = `...`;  // template literal, flush-left lines
```

Type check that keeps the pair honest at compile time (not just at test time):
`STORY_CONTRACT_FIELDS`'s element type is constrained against the generated `StoryDraft` keys via
`satisfies readonly (keyof StoryDraft)[]` — a schema rename that misses the constant fails
`tsc --noEmit`, before any test runs. (`StoryDraft` import is type-only; the core stays
addon-free.)

## 3. `src/baml/decompose.test.ts`

### 3a. Fixture surgery

- `CANNED`'s single story gains all five fields, populated with short S-066-01-flavored prose.
  `OPEN_MODEL_CANNED` wraps CANNED's bytes, so the provider-agnostic proof at index [4] extends to
  the five fields with zero edits.
- New `PARTIAL_CANNED`: CANNED's plan with the story carrying `scope`/`storyAcceptance`/
  `waveRationale` only (built by JSON surgery on a shared object, not string duplication).
- New `SHELL_CANNED`: the story with none of the five — the T-066-01-02 refusal shape.

### 3b. Op batch — append, never reorder

`RESULTS` indices [0]–[4] are load-bearing (research §5). Append:
- `[5] { mode: "parse", text: PARTIAL_CANNED }`
- `[6] { mode: "parse", text: SHELL_CANNED }`

### 3c. New assertions

- Extend the existing `[0]` round-trip test (or a sibling test in the same describe): the five
  populated fields survive parse verbatim.
- New `describe("DecomposeEpic — story contract fields")`:
  - `[5]`: story present (not dropped), 3 present fields verbatim, `honestBoundary`/`outOfSlice`
    each `(x ?? null) === null` and `typeof x !== "string"` (typed absence, no default).
  - `[6]`: story present, all five absent — iterated over `STORY_CONTRACT_FIELDS`.
- New `describe("DecomposeEpic — render demands the story contract")` on existing render result
  `[2]`: prompt contains every `STORY_CONTRACT_FIELDS` name; prompt contains
  `STORY_CONTRACT_EXEMPLAR` (fallback per design D2: per-line trimmed containment); prompt
  contains the section's demand phrase (e.g. "story is a CONTRACT").
- Imports: `STORY_CONTRACT_FIELDS`, `STORY_CONTRACT_EXEMPLAR` (value imports from the addon-free
  core — permitted; the ban is on `baml_client` value imports only).

## Interfaces exposed to sibling tickets (the settled shape)

- `StoryDraft.scope | storyAcceptance | honestBoundary | waveRationale | outOfSlice`:
  `string | null | undefined` — absence is typed; presence is prose.
- `STORY_CONTRACT_FIELDS` / `StoryContractField` from `decompose-epic-core.ts` — T-066-01-02's
  gate iterates this; T-066-01-03's materializer maps field → section heading in its own file.
- `STORY_CONTRACT_EXEMPLAR` — T-066-01-02's tests may mine it for a passing fixture's prose.

## Ordering of changes

1. `decompose.baml` schema fields → `bun run baml:gen` (types exist before any TS references them)
2. `decompose-epic-core.ts` constants (typecheckable immediately via `keyof StoryDraft`)
3. `decompose.baml` prompt section (embedding the exemplar authored in step 2)
4. `decompose.test.ts` fixtures + ops + assertions
5. `bun run check` (gen → tsc → full suite)

Steps 2–3 are one authored unit (the exemplar is written once, pasted twice); the ordering above
is the verification order, and the whole ticket is small enough for a single atomic commit if
steps 1–4 prove out together (plan.md decides).
