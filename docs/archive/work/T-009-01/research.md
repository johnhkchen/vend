# T-009-01 — Research: `baml_src/propose.baml` (ProposeEpic → EpicCard)

Descriptive map of what exists and how it connects. No solutions proposed here.

## The ticket in one line

Author **ProposeEpic** as BAML — *authoring-only* (render via `b.request`, SAP-parse
via `b.parse`; BAML is never the transport). Add `baml_src/propose.baml` defining
`function ProposeEpic(signal, charter, project) -> EpicCard`, where `EpicCard` mirrors
the epic card: frontmatter (`id`, `title`, `kind`, `advances[]`, `serves`), the card
stat-block (`manaCost`, `color`, `type`, `rarity`), and the body (`intent`, `value`,
`doneLooksLike`, `context`). Plus offline unit tests (parse + render pins) and a
regenerated client.

## The proven pattern this ticket mirrors (DecomposeEpic, then CaptureNote)

Two plays already exist on exactly the substrate this ticket extends. The work is a
*third instance* of a settled pattern, not a new mechanism.

### BAML side — `baml_src/`
- `clients.baml` — `client<llm> ClaudeStub` (provider `anthropic`, model
  `claude-opus-4-8`, `api_key env.ANTHROPIC_API_KEY`, `max_tokens 32000`). **Never
  called.** Exists only to shape `.request`. Render-only call sites set a dummy key.
  ProposeEpic will reuse this same client verbatim.
- `decompose.baml` — `function DecomposeEpic(epic, charter, project) -> WorkPlan`.
  Establishes: the closed-set **enum poka-yoke** (`DraftType/Status/Priority/Phase`,
  each member uppercase-first with `@alias("lowercase-token")`); the `prompt #" … {{ x }}
  … {{ ctx.output_format }} "#` shape; the authored-judgment framing rendered into the
  prompt (the "clearing function" preamble). `@alias` is both what `{{ ctx.output_format }}`
  shows the model **and** what the SAP parser accepts on input; `b.parse` returns the
  enum **MEMBER** name.
- `note.baml` — `function CaptureNote(topic, project) -> Note`. The minimal second
  play: `class Note { title string; summary string; points string[] }`. Proves a play
  with **required scalar** fields behaves differently from WorkPlan under SAP (below).
- `generators.baml` — `generator typescript { output_dir "../" version "0.222.0" }`.
  Emits `baml_client/` at repo root. **`baml_client/` is gitignored** — a build product,
  regenerated by `bun run baml:gen`, never committed.

### Bridge side — `src/baml/`
The BAML native addon allows **exactly one** successful native call per `bun test`
process (its once-driven runtime reactor hangs the second). So tests never call
`b.request`/`b.parse` directly — they spawn a standalone child `bun` script that batches
all ops and returns JSON on stdout.
- `decompose-bridge.ts` — exports the **pure, play-agnostic** `extractPromptText(req)`
  (reaches into `req.body.json().messages`, keeps `type === "text"` blocks, joins). Also
  `runOp` for decompose. `if (import.meta.main)` guard sets a render-only key and runs the
  stdin→stdout protocol.
- `note-bridge.ts` — the sibling for CaptureNote. **Imports `extractPromptText` from
  decompose-bridge** (does not re-implement it). Defines `NoteBridgeOp`/`NoteBridgeResult`
  and a `runOp` with `parse`/`render` modes. This is the exact template for a
  `propose-bridge.ts`.

### Test side — `src/baml/*.test.ts`
- All BAML imports are **type-only** (a value import would load the native addon into the
  test process and reintroduce the flake). Enum fields are asserted against their
  string-literal member names (e.g. `"Task" as DraftType`).
- `runBridge(ops)` spawns `bun run <bridge>`, writes `{ ops }`, reads `{ results }`.
- One spawn per suite covers every op (native-addon limit is per process); ops batched
  in a module-level `RESULTS` promise, asserted in `test`s.
- `decompose.test.ts` pins: canned reply → typed WorkPlan; malformed → **empty** plan
  (WorkPlan is all-array, SAP never rejects → degrades). Render pin asserts the three
  inputs + the framing land in the prompt.
- `note.test.ts` pins: canned → typed Note; **garbage → REJECTED** (`b.parse` throws
  "required field" because Note has required scalars); present-but-empty → degrades to
  empty Note; render pin asserts topic/project/framing.

## The EpicCard target shape — sources

The output type must mirror the actual epic card. Authoritative sources:
- `docs/active/epic/TEMPLATE.md` — frontmatter (`id`, `title`, `status`, `advances[]`,
  `serves`) + body sections *Intent / Value to the design / Done looks like / Context &
  constraints*.
- `docs/active/epic/E-009.md` — a concrete, hand-cast card: frontmatter `id`, `title`,
  `status`, `kind: permanent`, `advances: [P1]`, `serves: >`; a stat-block ASCII card
  carrying `manaCost {2}{U}`, the type line "Permanent — Engine play", and
  "Rarity: Rare"; then *Intent / Value / Done looks like / Context & constraints*.
- `docs/knowledge/card-model.md` — the card vocabulary: `card = { name, manaCost,
  color (WUBRG discipline), typeLine (Sorcery=single-use | Permanent=reusable), text,
  rarity (common→mythic) }`. **Multi-color is normal** (DecomposeEpic = Azorius WU).
- `docs/knowledge/playbooks/propose-epic.md` — the play's rule-set `PE-1…PE-7` and the
  three gates. **PE-3** (assign a card: manaCost / color / type / advances) and **PE-4**
  (state value + an epic-level done-signal) are exactly the stat-block + `serves`/
  `doneLooksLike` fields the ticket calls out. **PE-1** (pull-only, the anti-overproduction
  guard) is the load-bearing framing to render into the prompt.

### Closed sets visible in the card model (candidate enums)
- **color** — WUBRG: White / Blue / Black / Red / Green. Multi-color is normal.
- **type / kind** — the single-use vs reusable axis: Sorcery | Permanent. The card carries
  this twice: frontmatter `kind:` (lowercase) and the stat-block type line (title-case) —
  the *same* axis rendered two ways (E-009: `kind: permanent` + "Permanent — Engine play").
- **rarity** — Common / Uncommon / Rare / Mythic (leaf → keystone).

`manaCost` is free-form mana notation (`{2}{U}`) — a string, not a closed set.
`id`, `title`, `serves`, and the four body fields are free strings; `advances` is a
string[] of charter-invariant ids (e.g. `["P1"]`).

## Tooling / commands
- `bun run baml:gen` → `baml-cli generate --from baml_src` (regenerates `baml_client/`;
  refuses on a version mismatch vs the pinned `0.222.0`).
- `bun run check:typecheck` → `tsc --noEmit`.
- `bun run check:test` → `bun test`.
- `bun run check` chains gen + typecheck + test.
- `check:committed` gate (E-008) runs on stop; source must be committed. `baml_client/`
  is gitignored, so regeneration does not dirty the tree.

## Constraints & assumptions
- **Authoring-only.** No live model call anywhere in this ticket. Tests are offline,
  render-only key.
- **`baml_client/` is generated, not committed.** Regenerate; do not stage it.
- **Native-addon one-call limit** dictates the subprocess-bridge + type-only-import
  discipline. Any deviation reintroduces test flake (observed historically).
- **SAP leniency depends on field shape.** EpicCard has required scalar fields
  (`id`, `title`, …) like `Note` — so a garbage reply should be **rejected** (throws),
  while a present-but-empty reply degrades. This is the behavior to pin, mirroring
  `note.test.ts`, not `decompose.test.ts`.
- Semantic validity of the card (id disjoint from the board, advances actually holds,
  non-goals respected) is **out of scope here** — BAML owns SHAPE; the PE gates
  (T-009-02) own MEANING. This ticket stops at the typed shape + render/parse pins.
