# T-009-02 — Research: ProposeEpic pure core (PE gates + card renderer)

Descriptive map of what exists and what this ticket plugs into. No solutions here —
those are Design's job.

## What this ticket is

The **pure core** of the ProposeEpic play: a `clear(card, ctx) -> GateVerdict` running
the three PE gates (value → bounds → structural) and a pure `renderCard(card) -> string`
that shapes an `EpicCard` into the `E-0XX.md` markdown. No engine, no fs, no BAML addon —
the addon-free mirror of `materialize.ts`/`note-core.ts`. T-009-01 already shipped the
BAML half (the `EpicCard` type + `ProposeEpic` function); T-009-03 will register the play
and add the impure card-writing effect. This ticket is the testable judgment in between.

## The type we judge — `EpicCard` (generated, T-009-01)

`baml_client/types.ts` (gitignored build artifact, regenerated by `bun run baml:gen`):

```ts
export interface EpicCard {
  id: string; title: string; kind: CardType; advances: string[]; serves: string;
  manaCost: string; color: CardColor[]; type: CardType; rarity: CardRarity;
  intent: string; value: string; doneLooksLike: string; context: string;
}
enum CardColor  { White="White", Blue="Blue", Black="Black", Red="Red", Green="Green" }
enum CardType   { Sorcery="Sorcery", Permanent="Permanent" }
enum CardRarity { Common="Common", Uncommon="Uncommon", Rare="Rare", Mythic="Mythic" }
```

Key facts (from `baml_src/propose.baml` + obs 20268/20800):
- Enum **member values are title-case strings** matching the member name (`b.parse`
  returns `"Blue"`, `"Permanent"`, `"Rare"`). The lowercase card-model tokens
  (`blue`, `permanent`, `rare`) are the `@alias` values — what the model SEES and what
  the renderer must EMIT into frontmatter/stat-block. So this ticket owns a
  **member→alias** map exactly like `materialize.ts`'s `TYPE_ALIAS`/`STATUS_ALIAS`.
- `kind` and `type` are the **same `CardType` axis rendered twice** (frontmatter
  `kind: permanent` vs. stat-block `Permanent`); T-009-01 design D3 kept both fields and
  noted the structural gate "can later assert `kind === type`".
- `color` is an **array** (multi-color is normal — Azorius = `[White, Blue]`). Frontmatter
  on disk (E-009) does *not* carry color; the stat-block does.
- EpicCard has **required scalars**, so SAP **rejects** a garbage reply (unlike all-array
  `WorkPlan` which degrades to empty). A present-but-empty reply coerces to blank-stringed
  fields — which the value/structural gates must classify as STOP (note.baml SAP note).

## The gates to implement — `playbooks/propose-epic.md` (PE-1…PE-7) §"The gates"

The playbook names three gates, one level up from DecomposeEpic's four:
1. **Value** — the card traces to a real signal and names a charter value it serves;
   advances-nothing-nameable is refused.
2. **Bounds** — no non-goal violated; prerequisites named; the `advances` holds.
3. **Structural poka-yoke** — valid card frontmatter (`id`, `advances`, `serves`, `kind`)
   and the `id` is **disjoint from the live board** (the R6/E-004 discipline at epic
   granularity).

A failed gate stops the line and says why (jidoka) — the andon-on-first-failure contract.

## The reuse target — `src/play/id-guard.ts`

```ts
export function detectCollisions(generated: readonly string[], existing: readonly string[]): string[]
```
PURE and TOTAL: returns the ids in `generated` that already exist in `existing`, deduped
and first-appearance ordered; `[]` means clear. Takes plain string arrays (no BAML import
at all — "the purest module in the tree"). The structural gate feeds it `[card.id]` against
the live epic ids. The same E-004 guard `materialize.ts` uses, now at epic granularity.

## The verdict shape — two candidate types

- `src/gate/gates.ts` declares `GateResult = GateClear | GateStop` (DecomposeEpic-specific
  `GateName` union). The ticket cites this for "the `GateResult` shape": a discriminated
  union `{status:"clear", cleared} | {status:"stop", gate, unit, reason}`.
- `src/engine/play.ts` declares `GateVerdict` — the **play-agnostic** version
  (`gate: string`, not the four-name union). Its doc-comment states gates.ts's `GateResult`
  is *structurally assignable* to it. `Play.gates` requires `GateVerdict`.
- **The note-core precedent**: `clearNote` (the second play, the one E-009 says to mirror)
  returns engine `GateVerdict` directly — so `note.ts` registers `gates: clearNote` with no
  adapter. This is the live pattern for a registered play's pure gate.

## The render target — the `E-0XX.md` epic card

`docs/active/epic/E-009.md` (real) + `TEMPLATE.md` show the shape a card materializes to:
- **frontmatter**: `id`, `title`, `status` (`open|clearing|active|done`), `kind`
  (lowercase alias), `advances: [P1]` (YAML flow array), `serves: >` (folded one-liner).
  Note: **no `color`/`manaCost`/`rarity` in frontmatter** — those live in the stat-block.
- a **card stat-block** (E-009 uses an ASCII box: name + `{2}{U}` mana, the
  `Permanent — …` type line, the serves prose, `Rarity: Rare`).
- the **body**: `## Intent — the bigger-picture play`, `## Value to the design`,
  `## Done looks like`, `## Context & constraints`.

## Render-pattern precedents

- `materialize.ts` `renderTicketFile`/`renderStoryFile`: PURE, array-join lines, `flowArray`
  for YAML lists, `alias(table, member, field)` that **throws `RangeError` on an unknown
  member** (enum/map drift = programmer error). Member→alias maps are module constants.
- `note-core.ts` `renderNoteFile`: PURE, returns `{name, body}`, deterministic prose,
  a trailer naming the play ("Captured by Vend's `capture-note` play"). Its test asserts
  the body `toContain`s each field — the "round-trips the card fields" pattern this AC wants.

## The clearing-context shape — what the gates need beyond the card

`gates.ts`'s `clear(plan, ClearContext)` takes `{epic, charter}` and **greps the charter**
at call time for live `P\d+`/`N\d+` ids (`matchIds`) — alignment recomputed, never stored
("a detectable defect, not silent rot", charter §"How planning uses this charter"). The PE
bounds gate needs the same charter grep. The structural gate needs the **existing epic ids**
(from `docs/active/epic/*.md`, gathered impurely by T-009-03 via `project-context.ts`'s
`listIdsIn`, exactly as materialize gathers board ids). So the pure `clear` takes a small
context: `{ charter, existingEpicIds }`.

## Charter invariants/non-goals the bounds gate greps (`docs/knowledge/charter.md`)

Invariants `P1–P7` (author-once, two-gesture, gates-are-contract, autonomy, local-first,
executor-agnostic, budget-hard). Non-goals `N1–N4` (not-a-copilot, not-a-dashboard,
not-a-prompt-runner, not-an-executor). The bounds gate: an `advances` entry shaped `N\d+`
(or matching a live non-goal) is incoherent ("cannot advance a non-goal"); a `P\d+`-shaped
entry not in the live charter is a dangling ref. Free-text `advances` (epic-outcome prose)
is human-judgment, not failed by rule — exactly `boundsGate`'s policy.

## The purity discipline (house pattern — obs 20402, 20532)

Every play splits a PURE core from an IMPURE shell because the orchestrator value-imports
`b` (the BAML native addon), whose once-driven reactor makes `bun test` flaky. This
module's **only** runtime imports are `detectCollisions` (pure) and node `path`-free logic;
the `EpicCard`/`CardColor`/… imports are **TYPE-ONLY** (erased under `verbatimModuleSyntax`),
so `propose-core.test.ts` loads no addon — the gates.ts/materialize.ts/note-core.ts rule.

## Verification surface

`bun run check:*` = `baml:gen` (so `EpicCard` resolves) + `tsc --noEmit` + `bun test`.
The test must be an ordinary pure-function test (no addon). AC demands four pins: a passing
card clears; a non-goal-violating card → bounds STOP; a colliding id → structural STOP; the
rendered markdown round-trips the card fields.

## Constraints / assumptions surfaced

- The original demand **signal is an input, not a field on the card** — so "the card names
  a signal it clears" can only be checked purely as *presence of value-bearing fields*
  (`serves`, `advances`, `value`), not as a literal back-reference. Semantic
  signal-tracing is human/model judgment, out of a pure gate's reach (Design must decide).
- `kind === type` agreement is checkable purely and is a natural structural check.
- The id-mint ("compute the next free `E-0XX`") is a pure helper this module exports for
  T-009-03's effect to use; the gate's job is the *disjointness check*, not assigning ids.
- `baml_client/` is gitignored — only `src/play/propose-core.ts` + its test + artifacts are
  committed; `check:committed` stays green (T-009-01 D7).
