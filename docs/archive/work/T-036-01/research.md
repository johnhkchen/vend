# T-036-01 — Research: open-model BAML client

Descriptive map of the territory this ticket touches. No solutions here — that is `design.md`.

## What the ticket asks

Two deliverables, both *authoring-layer only* (no live dispatch):

1. Declare an **`OpenModelStub`** BAML client (`provider openai-generic`) alongside `ClaudeStub`,
   env-driven, render-only, `baml:gen` green.
2. Extend the **render bridge** so a render op can target a **selectable client** and prove the
   built request is in **openai-generic format** (assert on request *shape*, not prompt text).

The dispense itself rides E-035's `Executor` seam, not BAML. BAML in Vend is for AUTHORING only.

## The BAML authoring layer as it stands

### `baml_src/` — the source of truth

Eight `.baml` files. Relevant here:

- `clients.baml` (15 lines) — declares the single client today, **`ClaudeStub`** (`clients.baml:8`).
  - `provider anthropic`; options `model "claude-opus-4-8"`, `api_key env.ANTHROPIC_API_KEY`,
    `max_tokens 32000`.
  - A 7-line header comment establishes the **render-only discipline**: "We never actually CALL
    this client." `b.request.*` renders the prompt; `b.parse.*` SAP-parses the reply; the dispense
    rides `src/executor/claude.ts`. The `api_key` is read only to BUILD the request object and is
    never sent — call sites set a dummy key. This comment is the pattern `OpenModelStub` mirrors.
- `generators.baml` — one `generator typescript`, `output_dir "../"` ⇒ client lands at repo-root
  `baml_client/` (gitignored, regenerated by `bun run baml:gen`, never committed). Pins
  `version "0.222.0"`; a CLI mismatch makes generate refuse.
- `decompose.baml`, `expand.baml`, `note.baml`, `propose.baml`, `steer.baml`, `survey.baml` — one
  function each. Each defaults to `ClaudeStub` (no per-function client override today).

### `baml_client/` — the generated client (build product, gitignored)

14 files. The pieces this ticket reads:

- `sync_request.ts` / `async_request.ts` — `b.request.*` builders. Each function takes its typed
  inputs plus an optional `__baml_options__?: BamlCallOptions`. **The key contract** (sync_request.ts:29):
  ```ts
  type BamlCallOptions = { tb?; clientRegistry?: ClientRegistry; client?: string; env?; events? }
  ```
  And the body of every builder resolves it (sync_request.ts:51-56):
  ```ts
  let __clientRegistry__ = __baml_options__?.clientRegistry;
  if (__baml_options__?.client) {                      // a plain string client name…
    __clientRegistry__ = __clientRegistry__ || new ClientRegistry();
    __clientRegistry__.setPrimary(__baml_options__.client);   // …becomes the primary client
  }
  ```
  So **runtime client selection has two surfaces**: pass `clientRegistry` (a `ClientRegistry` you
  built with `addLlmClient`/`setPrimary`), OR pass `client: "OpenModelStub"` (a bare name BAML
  resolves against the declared clients). The string form is the ergonomic one and needs no import.
- `sync_client.ts` — `b.request` / `b.parse` namespaces (the bridge imports `{ b }` from here).
- `native.d.ts` — `HttpRequest` exposes `.url`, `.method`, `.headers`, `.body` where
  `body.json(): any` returns the parsed provider payload. `ClientRegistry` exposes
  `addLlmClient(name, provider, options, retryPolicy?)` and `setPrimary(primary)`.

### `src/baml/` — the bridge pattern (per-function)

Each BAML function has a `*-bridge.ts` + `*.test.ts` pair (decompose, expand, note, propose,
steer, survey). The canonical one this ticket extends is **`decompose-bridge.ts`**:

- **Why it exists** (header, lines 1-15): the BAML native addon allows exactly **one** successful
  native call per `bun test` process (its async runtime reactor is driven by bun's loop only once;
  a 2nd `b.request`/`b.parse` hangs to timeout). A plain `bun` *child* process has no such limit.
  So tests spawn the bridge as a child, batch ops over stdin JSON, read results from stdout JSON.
- **`BridgeOp`** (`:21`) — a discriminated union: `{ mode:"render", epic, charter, project }` or
  `{ mode:"parse", text }`.
- **`extractPromptText(req)`** (`:35`) — PURE. Reaches into `req.body.json().messages`, flattens
  each message's `content` (array of blocks **or** a scalar string — handles both), keeps
  `type:"text"` blocks, joins. Narrowly typed because BAML does not publicly type the body.
- **`runOp(op)`** (`:47`) — render path calls `b.request.DecomposeEpic(epic, charter, project)`,
  casts the result to reach `.body.json().messages`, returns `extractPromptText(req)`.
- **Entry point** (`:63`, `import.meta.main`) — sets `ANTHROPIC_API_KEY ??= "baml-render-only"`
  (the render-only key guard), reads `{ ops }`, maps `runOp`, writes `{ results }`.

### `src/baml/decompose.test.ts` — the test harness

- All BAML imports are **TYPE-ONLY** (a value import would load the native addon into the test
  process and reintroduce the once-driven flakiness — observed 2026-06-18).
- `runBridge(ops)` (`:31`) spawns `bun run decompose-bridge.ts`, pipes ops in, parses results.
- One spawn covers every op (native limit is per-process), so the suite batches a parse + a
  malformed-parse + a render into a single `RESULTS` promise and asserts on each.
- The render test (`:146`) asserts the prompt **contains** the sentinel inputs and the authored
  "clearing function" framing — i.e. it asserts on rendered *text*, not request *shape*.

## E-035 — the dispatch layer this mirrors (the provider shape source)

`src/executor/openai-compat.ts` (T-035-02) is the live OpenAI-compatible executor behind the
`Executor` interface. It is the authority for the *shape* `OpenModelStub` must produce:

- Env names (the ones this ticket reuses verbatim): `VEND_OPENAI_BASE_URL` (`:28`),
  `VEND_EXECUTOR_MODEL` (`:30`), optional `VEND_OPENAI_API_KEY` (`:32`); default base URL
  `http://localhost:11434/v1` (local Ollama, `:34`).
- `buildChatRequest` (`:102`) builds `POST {base}/chat/completions`, body
  `{ model?, messages:[{role,content}], stream, stream_options }`. The openai-generic shape:
  `/chat/completions` endpoint, flat string `content`, no `max_tokens`.
- `select.ts` registers it under id `"openai-compat"`; `VEND_EXECUTOR` selects it. The render
  client (this ticket) and the dispatch executor (E-035) share env names but are independent
  seams — T-036-02 will make render *follow* `VEND_EXECUTOR`.

## Spike (de-risking the one external contract) — observed shapes

Temporarily declared `OpenModelStub` (openai-generic, env base_url/model/api_key), ran `baml:gen`
(green, "Wrote 14 files"), and rendered `DecomposeEpic` against each client:

| field | default (ClaudeStub / anthropic) | `{ client:"OpenModelStub" }` (openai-generic) |
|---|---|---|
| `req.url` | `https://api.anthropic.com/v1/messages` | `http://localhost:11434/v1/chat/completions` |
| body keys | `model, max_tokens, messages` | `model, messages` |
| `body.max_tokens` | `32000` | *(absent)* |
| `messages[0].role` | `user` | `system` |
| `messages[0].content` | **blocks[]** (array) | **string** (scalar) |

Both shapes are cleanly distinguishable on the request body alone (no prompt-text contrivance
needed). `extractPromptText` already tolerates both `content` forms, so the rendered text can be
identical across clients — exactly as the ticket anticipates.

## Constraints & assumptions surfaced

- **No live dispatch.** Render-only; never CALL the client; don't touch SAP-parse (T-036-02).
- **Back-compat is byte-identical.** `ClaudeStub` and every function's default client unchanged;
  the no-option render must produce the exact same request it does today.
- **openai-generic needs a base_url to render.** `provider anthropic` has a built-in default
  endpoint; `openai-generic` does not — the render fails without `VEND_OPENAI_BASE_URL` present.
  So any render-against-`OpenModelStub` call site must set render-only env defaults, the same way
  the bridge already sets a dummy `ANTHROPIC_API_KEY`.
- **The native-addon one-call limit governs test design** — everything goes through the subprocess
  bridge; BAML imports in test files stay type-only.
- **`baml_client/` is gitignored** — the contract is verified against generated code but only
  `baml_src/` + `src/` changes are committed; CI runs `baml:gen` first (`check` script).
