- `interface DebitResult { wallet: Wallet; overshoot: Budget }` ŌĆö overshoot is the

## Open concerns / notes for the reviewer

1. **Glyphs in `formatWallet`** (`ŌŚć` / `ŌÅ▒`): chosen to match the IA-8 symbols used in the
   information-architecture doc. The exact production-line rendering is T-024-03's call;
   `formatWallet` returns a plain string the gesture can restyle/relayout. Low risk, but
   worth a glance if the TUI has a house glyph set I haven't matched.
2. **`debit` discrimination (`Usage | Budget`)**: narrows on a numeric `timeMs`. A
   `Budget` always has one; a `Usage` never does. This is robust for the two intended
   shapes, but if a future caller passes a malformed object with a stray `timeMs`, it
   would be read as a Budget. The intended call sites (T-024-02) pass well-typed values,
   so this is a documented assumption, not a live risk. Covered by the all-absent-Usage
   test.
3. **`canAfford` does not validate `predicted`**: a non-finite predicted fails the `<=`
   and returns `false` (safe-refuse) rather than throwing. Deliberate ŌĆö the loop should
   never feed NaN, and refusal is the correct fail-safe direction. Noted in a code
   comment; not separately unit-tested (the comparison semantics are language-guaranteed).
4. **The `funded`/`remaining` positive-int asymmetry**: `allocate` forbids a 0 dimension
   but `debit` lets `remaining` reach 0. This is the one deliberate divergence from
   budget.ts's `assertPositiveInt`-everywhere stance, documented in the module header and
   rationalized by IA-9/10 (depletion is a success state). Flagged here so a reviewer
   doesn't read it as an inconsistency bug.
