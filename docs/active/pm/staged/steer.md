## Forks

The genuine decisions only the human can make — each recommendation-first. Assent or override:

### Fork — Now that the articulation trilogy (expand E-016 / survey E-017 / steer E-018) has all landed, where do the next blocks point?
- **Why it matters:** Each path commits the next cycle to a different axis of vision-distance. The macro-wallet is the headline but trust-gated (E-014 HOLD); consolidation de-risks three plays shipped in a single day but closes little new distance; the surface opens a new segment before the counter mechanic exists; the DAG advances the core architecture but is several blocks with no user-facing payoff yet. The risk appetite for ungated autonomy — and whether to spend now on closure vs. frontier — is yours to set, not mine.
- **Options:**
  1. Consolidate & validate: live-steer sweep + cold-start envelope recalibration + the expand/survey/steer consistency probe — close out and de-risk the three just-shipped plays (and passively accrue E1 data).
  2. Chase the headline: run the E1 measurement sprint now to read the walk-away number and try to ungate the macro-wallet — the two-gesture autonomy capstone.
  3. Open the surface frontier: the design-language session + the graph-view human projection — make run/project state legible and reach the visual-thinking segment.
  4. Push the orchestration frontier: grow the 2-node chain into the typed multi-node DAG — the vision's graph-structured spine.
- **Vend recommends:** Consolidate & validate first. It is the cheapest, foundation-first move (R3): it de-risks the trilogy you just shipped, the consistency probe defends the core 'consistency' value prop on its newest and widest surface, and the live runs it generates passively accrue the very E1 data the wallet's gate needs — so it feeds the headline rather than competing with it. Then run the measurement sprint with real data behind it.

### Fork — How far do we take the inbound graph-view request, given it edges toward a Linear/SaaS integration the charter explicitly guards against?
- **Why it matters:** The signal is on-charter only as a one-way downstream render (Vend authors → the view mirrors). A two-way Linear sync is the one place P5 (local-first) and N2 (not a babysitting dashboard) rot — 'if a future proposal asks Linear to write back, it has left this signal.' How far we build now sets how close we sail to that line, and whether we open the visual-thinking segment before there is a TUI surface to host it.
- **Options:**
  1. B-slice only: render the existing decompose DAG as a local, one-way decomposition tree; defer the Mermaid export and any Linear adapter.
  2. A+B render: full local/Mermaid projection of tickets-as-data plus the decomposition tree; Linear explicitly out of scope for now.
  3. Defer entirely until a TUI/design epic is scheduled — the surface has no home to render into yet.
- **Vend recommends:** B-slice only. The staged signal itself names it the cheapest, highest-delight, lowest-risk slice — it renders work `decompose` already does, proves the segment with near-zero charter exposure (no export format, no adapter), and naturally informs the design-language session before any heavier increment is committed.
