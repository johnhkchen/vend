### Fork ‚Äî Once E-037 lands, what is the next frontier ‚Äî make vend shippable & learnable, or build more core capability?
- **Why it matters:** These are different axes, not different sizes: shippability vs. capability-depth vs. model-plurality. A tool no one can install or understand can't be validated in the real world, so it gates real feedback on everything else; but the DAG is the named v1 centerpiece and the vision's whole point. Picking wrong spends a multi-block frontier on the lower-leverage axis.
- **Options:**
  1. Distribution & onboarding (D-2 doctor ‚Üí D-1 delivery ‚Üí D-3 init ‚Üí X-1 hackathon example) ‚Äî make vend installable and the value legible
  2. Graph-structured orchestration (Frontier 3 multi-node DAG) ‚Äî build the v1 architectural centerpiece
  3. Live open-model runtime (Frontier 2) ‚Äî prove P6 with an open model autonomously clearing work
- **Vend recommends:** Distribution & onboarding. E-037 proves the loop works for real; the natural complement is making that proven loop available to real users. The grounding is fresh and de-risked (a green compile spike, two staged discovery docs ready to pull) and mostly adopts lisa's existing toolchain ‚Äî the DAG can follow once people can actually run and learn the tool.

X-1 tests that for near-zero cost on already-shipped plays. Building the bottle first is real work that pays off only if the onboarding value is actually there.
- **Options:**
