## The question

What does vend need to become installable like lisa ‚Äî `brew install johnhkchen/vend/vend`
‚Äî a self-contained, versioned, **local-first** (P5) delivery that lands a user at a working
two-gesture surface (P2), without rolling our own machinery? Three sub-questions fell out:
**how do we ship it**, **how does it self-verify** (`vend doctor`), and **how does a fresh
project get scaffolded** (`vend init`).

## Reference ‚Äî how lisa ships (the pattern to mirror, not reinvent)

lisa (Rust) ships through four off-the-shelf pieces, all of which have a vend-shaped analog:

| lisa | mechanism | vend analog |
|---|---|---|
| build ‚Üí bottle ‚Üí tap | **cargo-dist** (`dist-workspace.toml`, `tap = johnhkchen/homebrew-lisa`, shell+homebrew installers, GH Actions) | **`dist` in JS mode** (same tool ‚Äî see ¬ß6) |
| release ritual | **justfile** (`just release` ‚Üí tag `v${version}` ‚Üí CI publishes) | **a `justfile`** (we can bring one in) |
| runtime preflight | **`doctor.rs`** (`check_required_deps()` before `run_loop`) | **`vend doctor`** (see ¬ß5) |
| project scaffold | **`lisa init`** (CLAUDE.md + loop scaffolding) | **`vend init`** (the clearing-house layer ‚Äî see ¬ß7) |

## Dependency direction (load-bearing)

**vend depends on lisa; lisa does not require vend.** lisa is the lighter, standalone
loop-only flow; vend is the clearing-house layer that sits *on top of* a lisa project. This
one-way arrow drives three decisions:

- **`vend doctor` treats `lisa` as a required dependency** (alongside `claude`); a vend cast's
  execution capacity is "lisa and its agents" (charter core feature). *(Verified on this
  machine: `lisa 0.2.11` and `claude` both on PATH.)*
- **`vend init` runs *after* `lisa init`** and layers on ‚Äî it detects an existing lisa project
  (`CLAUDE.md` / `.lisa.toml`) and adds only the structures lisa didn't make. It never
  duplicates the lisa loop scaffold.
- The Homebrew formula can declare lisa as a dependency / `vend doctor` surfaces it as a fix-it,
  rather than vend trying to embed or replace it (N4 ‚Äî vend is not an executor).

## Runtime dependency map (what `vend doctor` checks)

Key finding from go-and-see: **the default Claude path needs no API key.** The live dispense
rides `claude -p` on the **subscription** (`src/executor/claude.ts`, `CLAUDE_CLI || "claude"`),
not BAML transport and not a metered key. The `ANTHROPIC_API_KEY` in every bridge is a
**render-only dummy** (`?? = "baml-render-only"`) ‚Äî read only to *build* the request object,
never sent. So doctor's required set is small and **executor-conditional**:

**Always required**
- `lisa` on PATH ‚Äî execution capacity (the dependency direction above).
- `claude` CLI on PATH + logged in ‚Äî the default executor (`claude --version`; honor `CLAUDE_CLI`).
- `git` ‚Äî the whole gate frame (committed/head/history + board-freshness) spawns git.
- `baml_client/` present ‚Äî a **gitignored build product** (regenerated by `bun run baml:gen`,
  pinned to baml-cli `0.222.0`); must be bundled in a package, never generated on the user's box.
- BAML native addon loadable ‚Äî a trivial in-process render proves it (see ¬ß5; spike says this
  embeds into a compiled binary).
- `bun` ‚Äî only if shipped as source/wrapper rather than fully compiled (the compile path embeds it).

**Only if `VEND_EXECUTOR=openai-compat`** (open-model path): `VEND_OPENAI_BASE_URL`,
`VEND_EXECUTOR_MODEL`, `VEND_OPENAI_API_KEY` (+ optional endpoint ping). These are *real*
(E-035's `OpenAICompatExecutor` calls them), unlike the Claude path's dummy.

**The trap to avoid:** a naive doctor demands `ANTHROPIC_API_KEY` and only passes with an
open-model endpoint set ‚Äî both backwards. Doctor must read `VEND_EXECUTOR` first and branch.

## BAML + the compile spike ‚Äî **PROVEN GREEN**

The make-or-break unknown was: can `bun build --compile` embed BAML's native `.node` addon?
Ran the spike this session and it passed:

- Wrote a throwaway entrypoint doing a real `b.request.DecomposeEpic(...)` **render** + a
  `b.parse.DecomposeEpic(...)` **parse**, compiled it with `bun build --compile`, and ran the
  resulting **~108 MB standalone binary from `/tmp`** ‚Äî a directory with **no `node_modules`
  and no `baml_client/`**.
- Result: `{"ok":true,"renderUrl":"https://api.anthropic.com/v1/messages","renderedMessages":1,
  "parsedStories":0,"parsedTickets":0}` ‚Äî render and parse both ran, fully self-contained.

Conclusion: **the compiled binary embeds both the BAML native addon and the generated client.**
Matches Bun's docs (it copies the `.node` to a temp dir and loads it; ~95% Node-API coverage).
Two supporting facts confirmed in source: BAML is **authoring-only** (never the transport ‚Äî the
only egress is `claude -p` or the open-model endpoint), and the "one native call per process"
hazard is a **`bun test`-only** artifact (worked around by the subprocess bridge in tests); a
normal `bun run` / compiled binary makes many calls in-process, which the spike's render+parse
demonstrates. The ~108 MB size is the Bun runtime + addon ‚Äî normal for compiled Bun binaries.

## Off-the-shelf tooling decision (don't roll your own)

- **Distribution ‚Üí `dist` (formerly cargo-dist), JS mode.** It added JavaScript support in
  0.21.0 and **drives `bun build --compile` itself**, then generates the **Homebrew formula/tap
  + shell installer + GitHub Actions CI** ‚Äî the whole pipeline vend lacks, in one tool.
  **lisa already uses it** (`cargo-dist-version` pinned), so this is one toolchain across both
  repos and the same `tap` pattern. Actively maintained (0.30.x through late 2025). *Caveat:*
  the JS path is newer than the Rust path ‚Äî needs a small `dist.js` build script rather than
  full zero-config, and its "npm installers" wrap prebuilt binaries (not a native npm publish).
  *Considered and rejected:* **oclif** ‚Äî a heavier CLI *framework* (restructures the command
  model, npm/tarball-first); vend already has a clean pure/shell `cli.ts`, so oclif is invasive
  and off-shape. `dist` wraps the build without touching vend's architecture.
- **Release ergonomics ‚Üí a `justfile`** (we can bring one in) ‚Äî mirror lisa's `just release` /
  `just check` / `just install` rituals on top of `dist` + the existing `package.json` scripts.
- **Doctor probing ‚Üí `envinfo`.** There is **no drop-in "doctor framework"** ‚Äî the mature CLIs
  (expo-doctor, react-native doctor, npm doctor) all hand-roll a thin check layer on top of
  **`envinfo`**, which does cross-platform binary/version/package detection. Adopt `envinfo`
  for the grunt work; hand-roll only the irreducibly vend-specific checks (lisa/claude reachable,
  `baml_client` bundled, BAML addon loads, the `VEND_EXECUTOR` branch). Wrap in vend's standard
  pure-core (`doctor-core.ts`, fully tested) + impure-shell (`doctor.ts`, the probes) split.

`dist` and `doctor` are **complementary, not redundant**: `dist` gets the binary *onto* the
machine; `doctor` verifies the machine can *run a cast*. lisa proves the split (cargo-dist for
distribution **and** `doctor.rs` for runtime deps).

## `vend init` ‚Äî the clearing-house scaffold (complement to `lisa init`)

`lisa init` makes the lighter loop-only project (CLAUDE.md + loop scaffold). `vend init` adds the
structures the vend + lisa combo needs that the bare loop doesn't ‚Äî **idempotent, layered onto an
existing lisa project:**

- `docs/active/demand.md` ‚Äî the pull board (template header; **starts honestly empty** per IA-4).
- `docs/active/pm/` ‚Äî the PM desk: `README.md`, `process-gate.md` (gate down), `staged/`.
- `docs/active/epic/` (+ `TEMPLATE.md`), `docs/active/stories/`, `docs/active/tickets/`,
  `docs/active/work/` ‚Äî the card hierarchy.
- `docs/knowledge/` ‚Äî `charter.md`, `vision.md`, `playbooks/` (steering context) ‚Äî seeded as
  stubs/anchors the author fills in.
- `docs/archive/demand-cleared.md` ‚Äî the cleared ledger.
- `.vend/` runtime state + `.gitignore` entries (telemetry ignored, `decisions.jsonl` kept);
  `.mcp.json` for the codebase-index MCP; `baml_src/` if authoring locally.

**Sharp design line (IA-3 / IA-4):** `vend init` seeds **structure and knowledge scaffolding,
never demand.** The board starts empty; the first move is a **Survey cast**, not fabricated
signals. Seeding demand would violate read-never-invent.

## Candidate signals (un-promoted ‚Äî a new frontier the board is silent on)

This is **distribution & onboarding**, absent from the six frontiers in `demand.md`. P5-aligned,
de-risked by the spike, and mostly *adopting lisa's existing toolchain* (low-risk for a new
frontier). Three coherent candidates, ~1‚Äì2 blocks total:

- **D-1 ¬∑ Homebrew delivery via `dist` (JS mode) + `justfile`.** Real `package.json` (`bin`/
  semver/drop `private`), `bun build --compile` across targets, tap + shell installer + release
  CI. *Advances P5; spike-proven.* **High.**
- **D-2 ¬∑ `vend doctor` preflight** on `envinfo` + the bespoke checks, wired as a **cast
  precondition gate** (lisa's `check_required_deps()`-before-`run_loop` move). *Advances P3/P5;
  IA-9 (a stop is successful refusal).* **High** ‚Äî small.
- **D-3 ¬∑ `vend init`** ‚Äî the clearing-house scaffold complementing `lisa init`; idempotent on an
  existing lisa project; seeds structure+knowledge, not demand. *Advances P5; IA-3/IA-4.* **High.**

Natural sequence: D-2 (doctor ‚Äî smallest, immediately useful for debugging installs) ‚Üí
D-1 (delivery) ‚Üí D-3 (init). D-1 and D-3 both want the `package.json`/`justfile` cleanup, so they
share a foundation.
