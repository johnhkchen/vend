# T-002-01-02 — Plan: test-gate-subclass-and-router

*Ordered, independently-verifiable steps + the testing/verification strategy.*

## Testing strategy (stated up front)

This ticket ships **no `*.test.ts`** — and that is correct, not a gap:

- The `Test` sub-class is a **trigger-and-report shell**. There is no logic to
  unit-test: the Central Rule keeps every assertion in the app's
  `bun run check:test`. A `/ci` unit test would have nothing to assert.
- A `*.test.ts` under `/ci` would risk being collected by the app's
  `bun test` (`src/**/*.test.ts` pattern lives at the repo root) — a boundary
  violation. (Even though `/ci` is outside `src/`, we add nothing test-shaped.)
- **Verification here is two-fold and structural:**
  1. **Compile** — the module compiles in its own toolchain (`tsc` green) once
     `sdk/` is generated (AC4).
  2. **Non-perturbation** — the app's own gates stay green and unchanged (the
     app build is untouched).
- The **behavioural** proof (`dagger call test run` actually runs the suite,
  *agrees* with standalone, and *can fail*) is **owned by T-002-01-03** and is
  intentionally deferred. Pulling it in is the over-build reflex.

## Steps

### Step 1 — Write the gate (`ci/src/test.ts`) and route it (`ci/src/index.ts`)

Author per Structure:

- `ci/src/test.ts` — `@object() export class Test` with one
  `@func() async run(source)`; container chain `oven/bun:1.3.9-slim` →
  mount source (`@argument` `defaultPath: "/"`, `ignore` node_modules/baml_client/.git)
  → `bun install --frozen-lockfile` → `bun run baml:gen` → `bun run check:test`
  → `.stdout()`. Header documents prep-vs-check.
- `ci/src/index.ts` — add `import { Test } from "./test"`, add `func` to the
  dagger import, add `@func() test(): Test { return new Test() }`, update header.

**Verify (no engine yet):** `grep -RnE "from \"\\.\\./|/src/|@boundaryml" ci/src`
→ empty (no app import). `index.ts` contains exactly one `@func()`. Files are
well-formed TS by eye.

### Step 2 — Generate `sdk/` (one deliberate `dagger develop`, guarded)

The andon-guarded codegen (Design Decision 2). Docker is up.

1. Snapshot: `git stash list`/`git status` clean baseline; record

2. Run `dagger develop` **for the `/ci` module** at the pinned engine.
3. **Andon check** — `git diff -- ci/dagger.json ci/package.json ci/tsconfig.json
   ci/.gitignore ci/.gitattributes`:
   - If `engineVersion` changed from `v0.21.4`, or any tracked `/ci` file was
     mutated beyond creating gitignored `sdk/` → **revert that change and surface
     it** in `progress.md` (do not accept a silent bump). 
   - Expected: no tracked-file diff; only `ci/sdk/` (gitignored) + maybe
     `ci/node_modules/` (gitignored) appear.

**Verify:** `ls ci/sdk/index.ts` exists; `git status` shows **no** modified
tracked `/ci` config files (only new `ci/src/test.ts` + edited `ci/src/index.ts`).

### Step 3 — Compile in the `/ci` toolchain (AC4)

Run the module's own typecheck:

- Ensure `typescript` is available in `/ci` (install if `dagger develop` didn't:
  `cd ci && bun install`), then `cd ci && bunx tsc --noEmit -p tsconfig.json`

**Verify:** exit 0 — `@dagger.io/dagger` resolves to `./sdk`, decorators compile,
`test.ts`/`index.ts` typecheck clean. If `tsc` reports unresolved
`@dagger.io/dagger`, `sdk/` is missing → return to Step 2.

### Step 4 — Prove the app build is untouched

From the repo root, unchanged:

- `bun run check:typecheck` → exit 0 (app `tsc` still `include: ["src"]`; never
  reaches `/ci`).
- `bun run check:test` → still **229 pass / 0 fail** (baseline identical).

**Verify:** both exit 0, pass count matches baseline; `git status` shows only
`ci/src/test.ts` (new), `ci/src/index.ts` (modified), and the work artifacts —
**no** app-source edits, no root `package.json`/`tsconfig.json` changes, no
`workspaces` key added.

## Risks & mitigations

- **`dagger develop` bumps the engine or mutates a tracked file.** → Step 2 andon:
  diff, revert, surface. Pin == CLI version, so a bump is unlikely.
- **`tsc` in `/ci` can't find a TS binary / sdk deps.** → `cd ci && bun install`
  first; `sdk/` brings what it needs. If still blocked, surface in `progress.md`.
- **Someone "fixes" the `@dagger.io/dagger` import by adding it to
  `package.json`.** → Review reiterates: it resolves via generated `sdk/`
  path-mapping, never an npm dep.
- **`baml:gen` needs network/binary in-container.** → not exercised here (no run);
  flagged for T-003. The gate's *code* is correct regardless; T-003 validates
  execution and adjusts prep if the linux `baml` binary needs more.
