# T-002-04 — Structure: live-dispense-proof

The blueprint. All apparatus lives under the ticket's work dir per the directory
conventions ("Work artifacts, one subdirectory per ticket ID"). **No `src/` file is
created, modified, or deleted** — this is a spike that runs already-built code.


(sandboxes), `.vend/transcripts/<runId>.jsonl`, `.vend/runs.jsonl` (the one ledger).

## `live-proof.ts` — the driver (impure apparatus)

A plain `bun` script run from the repo root. Imports the **real** orchestrator — no
doubles:

```ts
import { runDecomposeEpic, type RunSummary } from "../../../../src/play/decompose-epic.ts";
```

(4 `../` from `docs/active/work/T-002-04/` → repo root, then `src/…`.)

### Internal shape (top to bottom)

- **`REPO`** — `process.cwd()` (driver is always launched from repo root).
- **`prepSandbox(name): Promise<string>`** — impure. `rm -rf` then `lisa init --path
  <REPO>/.vend/live-proof/<name>` via `Bun.spawn`; copy `docs/knowledge/charter.md`
  into `<sandbox>/docs/knowledge/charter.md`. Returns the absolute sandbox path. This
  is the D2 isolation primitive.
- **`Scenario`** type — `{ id, ac, epicPath, budget, projectRoot, expect }` where
  `expect` is the anticipated `RunOutcome` (for shape-checking, not enforcement).
- **`SCENARIOS`** — the four rows from design D3, built after sandboxes are prepped.
- **`runScenario(s): Promise<ScenarioResult>`** — impure. Wall-clock wraps
  `runDecomposeEpic({ epicPath, budget, projectRoot, runId: s.id, transcriptDir:
  <REPO>/.vend/transcripts })`; catches nothing it shouldn't (a genuine launch failure
  surfaces). Returns `{ s, summary, wallMs, validatePassed? }` and reads the terminal
  `result` from the transcript to extract the **true model id** + `usage` +
  `total_cost_usd` (closing review concern #4 by reading, per design D5).
- **`main()`** — sequential (not parallel: subscription-friendly + readable logs):
  prep each sandbox, run each scenario, collect results, write
  `results/summary.json` and (for A1) `results/e001-machine-plan.md`, print a table.
  Runs **sequentially** so transcripts/logs interleave cleanly and credit use is
  visible run by run.
- **`if (import.meta.main) await main()`** — entry guard (house idiom).

### What the driver must NOT do

- Must not weaken or bypass a gate (design D4 / P3).
- Must not write to `docs/active/{stories,tickets}` of the repo (only sandbox roots).
- Must not pass a run-log path override — the cwd-relative default keeps one ledger.

## `fixtures/tiny.md` — minimal groundable epic (A2, A4)

A real-shaped but *small* epic so the wasted/near-free calls are cheap. Frontmatter
(`id`, `title`, `status`, `advances`, `serves`) + a short Intent / Done-looks-like
naming one concrete, groundable outcome against this project (e.g. a `vend runs`
pretty-printer over the existing run log). Its only job is to be a valid live input;
its outcome is decided by the budget, not its content.

## `fixtures/underspecified.md` — the gate-trip epic (A3)

Deliberately contentless: valid-enough frontmatter but an Intent/Done body that names
**nothing groundable** ("make it better; specifics TBD"). Per design D3/Research
hazard 2 this drives the SAP empty-degradation path → empty `WorkPlan` → **value**
gate stop. The fixture's job is to be honestly under-specified, not malformed.

## `results/summary.json` — machine evidence (generated, committed)

One object per scenario:
```jsonc
{ "id": "A1", "ac": "AC1+AC4", "outcome": "success", "expected": "success",
  "materialized": true, "validatePassed": true, "wallMs": 41234,
  "model": "claude-…", "usage": { "input_tokens": …, "output_tokens": … },
  "tokensCounted": 12345, "costUsd": 0.0 }
```
Plus a top-level `{ generatedAtNote, ledger: ".vend/runs.jsonl", runs: [...] }`.

## `proof.md` — the AC4 deliverable

The human note: a results table (tokens / cost / wall-clock per scenario), the
**E-001 by-hand vs machine diff** (story/ticket counts, id scheme, dependency shape,
where the gates would have caught a machine miss), confirmation of each AC, and the
**first kaizen signals** for the play (prompt/gate/fixture observations). This is the
file AC4 asks for; the RDSPI `review.md` summarizes it for the human reviewer.

## Ordering of changes

1. fixtures (`tiny.md`, `underspecified.md`) — inputs first.
2. `live-proof.ts` — the apparatus.
3. **live run** (`bun docs/active/work/T-002-04/live-proof.ts`) — generates
   `results/` + `.vend/` evidence. (This is the Implement phase's real work.)
4. `proof.md` — written from the generated evidence.
5. `progress.md`, then `review.md`.

## Verification surface

- `bun run check` stays green (no `src/` change ⇒ 114 tests unaffected; only a
  re-`baml:gen` + typecheck of the new driver under `tsc`).
- Each scenario's **actual** outcome matches `expect` (or the deviation is recorded).
- `.vend/runs.jsonl` gains exactly **4** countable lines (`wc -l`), one per run.
- A1's sandbox passes `lisa validate` (the runner logs `· lisa validate ✓`).
- `git status` shows only intended files (apparatus + write-ups); `.vend/` stays
  untracked.
