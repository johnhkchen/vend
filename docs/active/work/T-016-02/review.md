## How it works

`fragment → expand → signal` is now a registered, castable play and a one-gesture CLI surface:

```
vend expand "<rough fragment>"
   → castExpandFragment → castPlay(expandFragmentPlay, …)
        render  b.request.ExpandFragment(fragment, charter, project)
        dispense (claude -p, wall-clock + 12k token budget)
        parse   b.parse.ExpandFragment → Signal  (garbage → EMPTY_SIGNAL)
        gates   clear(signal, {charter})  — honest-empty → read-never-invent → value-link
        effect  expandFragmentEffect → docs/active/pm/staged/<slug>.md   (STAGED, not promoted)
        log     one run-log record
```

The headline AC — **staging, not promotion** — is enforced by `expandFragmentEffect`: it writes
ONLY under `docs/active/pm/staged/`, never `demand.md` or the board. The staged file carries the
signal in the `demand.md` row shape (`renderSignalRow`) plus the exact `vend chain "<what> — <why>"`
pull string a human uses to promote it — the README's "staging unit is a signal string" made
literal. An honest-empty / read-never-invent refusal STOPs at the gate (`gate-failed`) so the
effect never runs and nothing materializes — the fabricated-signal andon.

The play is the **fourth** registry entry casting through the same generic `castPlay` — zero
per-play branches in the engine; it mirrors `propose-epic.ts` one notch upstream, minus the id
mint (a signal has no DAG identity).

## Open concerns / flags for the human

1. **`budget` is still a model-proposed string, not the measured E-013 envelope.** Carried over
   from T-016-01's open concern #1. The staged row shows the model's rough `budget` (e.g.
   `small (~1h)`); wiring the measured `recalibrate` pre-fill into the staged signal is a deliberate
   follow-up, not this ticket's scope. Worth a conscious check when the macro-wallet work resumes.
2. **Overwrite-by-slug is intended, not a bug.** Two expands whose `what` slugs identically write the
   same `pm/staged/<slug>.md` — idempotent staging (re-expanding a fragment updates its draft). This
   diverges from `proposeEpicEffect`'s unique-id re-mint **on purpose** (a staged draft has no DAG
   identity); documented in the effect header. If a human wants to keep two same-titled drafts, edit
   one's `what` before re-expanding. No clock/random was introduced (keeps the effect deterministic
   + testable).
3. **`docs/active/pm/staged/` is a new dir created on first cast.** It does not exist until `vend
   expand` runs once (the effect `mkdir -p`s it). The PM README's handoff diagram describes
   `proposed-batch.md`; this adds a sibling machine-written inbox the README anticipated ("a future
   play could batch-read this file directly"). A doc touch-up to README naming the `staged/` inbox is
   a reasonable (non-blocking) follow-up.
4. **read-never-invent stays a presence check, not a groundedness oracle** (unchanged from T-016-01).
   A pure gate refuses a signal that cites *nothing*; whether the citation is *real* is the human's
   judgment at pull time (staging-only, PE-1). The honest limit of a pure gate.
