# T-066-01-01 — story-contract-schema-and-render — Research

Descriptive map of everything this ticket touches. No solutions proposed here.

## 1. The ticket in one line

Extend the decompose story type with five contract fields (`scope`, `storyAcceptance`,
`honestBoundary`, `waveRationale`, `outOfSlice`) end to end — BAML schema → generated types →
render prompt — with the render demanding the sections and embedding `docs/active/stories/
S-066-01.md`'s shape as the exemplar. A missing field must parse to a **typed absence**, never a
fabricated default. Downstream consumers (the gate, the materializer, the doc) are sibling tickets
T-066-01-02/-03/-04, all `depends_on` this one — this ticket settles the shape they consume.

## 2. Where the story type actually lives

The chain has three links; "the story type" is defined once and flows through:

1. **`baml_src/decompose.baml`** — the source of truth. `class StoryDraft` (lines 73–81) carries
   exactly six fields today: `id`, `title`, `type`, `status`, `priority`, `tickets` — no body
   content at all. `class TicketDraft` by contrast already carries a value triplet
   (`purpose`/`advances`/`doneSignal`), which is why tickets materialize with real bodies and
   stories materialize as shells. `WorkPlan` is the all-array wrapper (`stories[]`, `tickets[]`).
2. **`baml_client/`** — generated TypeScript, **gitignored** (a build product). Regenerated by
   `bun run baml:gen` (`baml-cli generate --from baml_src`, version pinned 0.222.0 in
   `generators.baml`). `bun run check` runs `baml:gen` first, so the generated types always track
   the `.baml` source. Current generated shape: `baml_client/types.ts:163` `interface StoryDraft`
   with the six required fields.
3. **`src/baml/decompose-bridge.ts`** — the subprocess render/parse bridge for tests. It does not
   define the type; it re-exports parse results typed as `WorkPlan` from `baml_client`. The ticket's
   phrase "extend the story type in `src/baml/decompose-bridge.ts` (and the `.baml` schema it
   bridges)" resolves to: the type change is made in `decompose.baml`, the bridge picks it up
   through the regenerated client with no structural change of its own.

## 3. Where the render lives

The prompt is the BAML template inside `function DecomposeEpic(epic, charter, project)` in
`baml_src/decompose.baml` (lines 89–132). It is a static authored template with three
interpolations (`{{ epic }}`, `{{ charter }}`, `{{ project }}`) plus `{{ ctx.output_format }}`,
which renders the output schema (field names, types, `@description` strings, enum aliases) to the
model. Two call sites render it:

- **Live path**: `decomposeEpicPlay.render` in `src/play/decompose-epic.ts:231` —
  `b.request.DecomposeEpic(i.epic, i.charter, i.project, …)` → `extractPromptText`.
- **Test path**: `src/baml/decompose-bridge.ts` `runOp` — same call, in a spawned child `bun`
  process (the native addon allows one call per `bun test` process; the bridge batches ops and
  runs them all in one child).

`src/play/decompose-epic-core.ts` (named by the ticket as the render's home) contains **no render
function today**. It is the addon-free pure core: outcome classification, stream formatting, and
the pure plan transforms (`stripNonGoalAdvances`, `renumberPlanToEpic`, `graphIntegrityViolations`,
`blockEntryTicketsAfter`). Its established role is *"constants and pure judgment the tests can load
without the BAML addon"* — `DECOMPOSE_MAX_TURNS` and `DECOMPOSE_TOOLS` live here for exactly that
reason. Any prompt-adjacent constant this ticket introduces has its natural home here.

## 4. Parse semantics — what "typed absence" means in this codebase

- `b.parse.DecomposeEpic(text)` is SAP (schema-aligned parsing). The suite already pins one
  leniency hazard (`decompose.test.ts` "malformed reply degrades to an EMPTY plan"): `WorkPlan` is
  all-array, so SAP never rejects — garbage degrades to `{stories: [], tickets: []}` and the value
  gate classifies empty as malformed.
- BAML field optionality: a `string?` field generates `field?: string | null` in the TS client and
  SAP yields `null` when the reply omits it. A **required** `string` field that the reply omits
  causes SAP to fail coercion of that object — and inside an all-array plan a story that fails to
  coerce is *dropped*, i.e. a silent disappearance rather than a visible absence. No `.baml` file
  in this repo uses optional fields yet; this ticket would be the first, so the parse behavior
  must be pinned by a fixture test, not assumed.
- The ticket's contract — "a story missing a field surfaces as a typed absence for the downstream
  gate to refuse" — combined with T-066-01-02's plan (a gate that STOPs on missing/empty sections,
  fixture = "today's ten-line shell") tells us the parse layer must **admit** an incomplete story
  and hand it to the gate, not reject or drop it.

## 5. Existing test patterns to follow

- **`src/baml/decompose.test.ts`** — the home for parse round-trip + render assertions. Pattern:
  one `runBridge([...ops])` spawn shared by every test (indices into `RESULTS` are stable;
  history appends new ops at the end to preserve them). `CANNED` is the canonical populated reply
  (aliases in, enum member names out). Render tests assert **containment** (`toContain`) of
  sentinel inputs and authored phrases — the repo uses no `toMatchSnapshot` anywhere; "snapshot"
  in this repo's tickets has meant "assert the rendered prompt carries the demanded content".
- **Type-only BAML imports** in test files (erased at runtime) — a value import of `baml_client`
  into a `bun test` process loads the native addon and makes the suite flaky. Enum values are
  compared as string literals cast to the type-only enums.
- `src/play/decompose-epic.test.ts:131` and `src/play/materialize.test.ts:49` build `StoryDraft`
  fixtures via `Partial<StoryDraft>` spreads with the six current fields — they typecheck
  unchanged iff the new fields are optional in the generated interface.

## 6. Downstream consumers that must keep compiling

- **`src/play/materialize.ts` `renderStoryFile`** (line 161) — reads `id/title/status/priority/
  tickets`, writes the one-line shell body. T-066-01-03 rewrites the body; this ticket must not
  break its compile. New optional fields are additive — untouched code keeps compiling.
- **`src/gate/gates.ts` `clear(plan, {epic, charter})`** — the gate list T-066-01-02 extends.
  Imports `StoryDraft` type-only. Additive optional fields: no impact.
- **`stripNonGoalAdvances` / `renumberPlanToEpic`** in the core spread stories with `{...s}` —
  additive fields flow through untouched.

## 7. The exemplar source

`docs/active/stories/S-066-01.md` is hand-authored (footer: "the look-and-feel exemplar for the
contract this story ships"). Its shape: a prose paragraph (epic framing), then **Scope**, **Story
acceptance**, **Honest boundary** as bolded lead-in paragraphs, a `## DAG` fenced block, a *Wave
rationale* paragraph, and **Out of this slice**. Note the DAG block is *derived* from ticket
`depends_on` edges by the materializer (T-066-01-03), not a sixth story field — the five fields
this ticket adds are exactly the prose sections.

## 8. Constraints and assumptions surfaced

- **BAML prompts are static templates** — they cannot import TS constants. Any exemplar text shown
  in-prompt must be authored inside `decompose.baml` (or threaded as a new function parameter,
  which would touch every render call site: the play, the bridge, and the bridge's op protocol).
- **`baml_client` is gitignored** — the schema change is reviewable only in `decompose.baml`; the
  generated diff never lands in git. `bun run check` regenerates before typecheck.
- **The `{{ ctx.output_format }}` block** will automatically show the five new fields (with their
  `@description`s, and their optionality) to the model — the prose demand and exemplar are
  *additional* pressure on top of the schema display, not the only channel.
- **One-spawn test discipline**: new bridge ops must append to the existing `runBridge` batch.
- The verification gate is `bun run check` (baml:gen → tsc → bun test); `bun run lint` is not a
  live script in package.json despite CLAUDE.md's convention note.
