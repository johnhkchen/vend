## Context

Make the *first* play a registry entry and route the runner through the engine —
proving the generalization is behaviour-preserving. `DecomposeEpic`'s render
(`b.request.DecomposeEpic` + `extractPromptText`), parse (`b.parse.DecomposeEpic`),
gates (`clear`), effect (`materialize` + `lisaValidate`), budget, and card (Blue
Permanent) become one `Play`. `runDecomposeEpic` is reimplemented as
`castPlay(registry.get("decompose-epic"), …)`, and the CLI + shelf dispatch look the
play up by name — **no per-play hardcoding left**.

**Shares `src/play/decompose-epic.ts`, `src/cli.ts`, `src/shelf/press.ts` with the
prior tickets / each other** → depends on T-007-02 (R4).

**Cites:** `src/play/decompose-epic.ts` (refactor target); `src/cli.ts` +
`src/shelf/press.ts` (dispatch — look up by name + `castPlay`);
`playbook-decompose-epic.md` (the play's gates/effect).

## Acceptance Criteria

- [ ] `DecomposeEpic` is registered as a `Play` (render / parse / gates / effect /
      budget / card all wired from the existing code).
- [ ] `runDecomposeEpic` is `castPlay` over that entry — **no behaviour change**; the
      play-specific logic now lives in the registry entry, not the runner.
- [ ] `vend run <play> …` and the shelf press **dispatch by name** through the
      registry + `castPlay` (no hardcoded `decompose-epic` branch).
- [ ] All existing tests pass unchanged; DecomposeEpic still renders → dispenses →
      gates → materializes → `lisa validate` exactly as before.
