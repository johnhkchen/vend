## Context

**Put the SVG work-graph board in the browser the designer already has open for the app.** The designer is
previewing the Astro/Cloudflare app; this renders the `vend svg` board (E-055/E-056 — glanceable status
columns, blocked edges in red) into that same browsing context, so understanding the work is *looking at a
picture beside the running app*, not reading code.

**Small wiring over the SHIPPED renderer (no new render):**
- Reuse `vend svg` (E-055/E-056, `src/present/svg-file.ts`) which writes `.vend/work-graph.svg`.
- Surface it in the Astro seed's preview — the lightest honest option: an Astro page/route (e.g.
  `/board`) that displays the generated `.vend/work-graph.svg` (served as a static asset, or `<img>`/inline
  `<object>`), plus a one-liner in the README/`package.json` script to (re)generate it (`vend svg`). So the
  designer opens the app and a `/board` view shows the work-graph beside it. (A pre-rendered static SVG is
  enough — no live websocket; keep it deterministic + dependency-light, matching the SVG's own discipline.)
- Keep one-way authority: the board is a read-only projection (E-055/E-056 already guarantee `vend svg`
  never mutates the graph); the Astro view only *displays* the file.

**Codebase-memory (use it — project coding-task suggestion):** confirm the `vend svg` seam + output path with
the **codebase-memory-mcp** (project `Volumes-ext1-swe-repos-vend`) via `search_code`:
- `search_code(pattern: "svg-file|projectionToSvg|work-graph.svg|vend svg|present", regex: true, mode: "full", path_filter: "^src/(present|cli)")` — the `vend svg` command + the `.vend/work-graph.svg` output path the Astro `/board` view displays.

**Cites:** `src/present/svg-file.ts` (`vend svg` → `.vend/work-graph.svg` — reused); `src/cli.ts` (the `svg`
command); `examples/templates/hackathon-seed/` (the Astro app — add the `/board` view + the regen script);
