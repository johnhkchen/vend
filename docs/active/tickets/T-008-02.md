## Context

Make the gate fire automatically. Wire `.lisa/hooks/on-stop.sh` to run
`bun run check:committed` (T-008-01) when a session stops, surfacing a clear
**uncommitted-code andon** (the offending paths) — so a session can't quietly mark a
ticket done with uncommitted code. The Central Rule holds: the hook only *invokes*;
the logic stays in the script.

**Design note (do not fight it):** this check reads the **host's git working tree**,
which a Dagger container can't see — so the enforcement point is a lisa hook, not a
`/ci` sub-class. That is correct, not a shortcut.

**Andon / honest scope (R11):**
- **Confirm lisa's hook semantics** — does a non-zero `on-stop` hook *block*, or only
  warn? Do not assume. If it only warns, that still catches the human/steering (it
  would have surfaced all three incidents); record that as the first-slice behavior,
  with hard-blocking as a follow-up.
- **Meta-risk:** editing `.lisa/hooks/` affects **future** loops, not the running one;
  the live hook-trigger verification is deferred to the next session-stop. Test
  `check:committed` standalone here; flag the deferred end-to-end check. Keep the hook
  script minimal and robust (a malformed hook must not wedge the loop).

**Cites:** `.lisa/hooks/on-stop.sh` (the hook); the lisa setup-guide (hook semantics);
`D-005` (the target); `T-008-01` (`check:committed`).

## Acceptance Criteria

- [ ] `.lisa/hooks/on-stop.sh` runs `bun run check:committed` at session stop and, on
      failure, surfaces a clear andon naming the uncommitted source paths.
- [ ] Whether a failing hook **blocks vs warns** is determined and documented (not
      assumed); the first-slice behavior is recorded, with hard-block flagged as
      follow-up if needed.
- [ ] A dirty-tree demo shows the andon fires; the hook is minimal/robust; the
      effect-next-loop nature is noted. `bun run check:*` stay green.
