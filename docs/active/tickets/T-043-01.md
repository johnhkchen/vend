## Context

**Make `proposeEpicEffect` idempotent on title so a retry can't orphan a card.** E-039's live sweep
minted the doctor epic twice — E-042 (logged, decomposed) + **E-041** (childless duplicate, same
title `vend-doctor-preflight`, never logged). Root cause (verified): `proposeEpicEffect`
(`src/play/propose-effect.ts:65`) re-mints `nextEpicId(live)` = **max+1** each run (design D2 —
TOCTOU-safe against clobbering), so a *second* run for the same proposal grabs the *next* slot:
board …E-040 → mint E-041 (write, then a timeout before logging) → retry → board has E-041 → mint
E-042 (same title). `detectCollisions` (E-004) passes both times — each minted a **fresh** id, never
a **reused** one. The collision guard catches id REUSE; it can't catch a fresh id for a duplicate
PROPOSAL. The stable identity across a retry is the **title** (the model's `card.id` is re-minted).

**The change — title-keyed idempotency, mirroring the E-004 shape:**

1. **Pure title-dedup in `src/play/id-guard.ts`** (beside `detectCollisions`): e.g.
   `findExistingByTitle(cardTitle: string, existing: ReadonlyArray<{id:string; title:string}>):
   string | null` — returns the existing epic id whose title matches (normalize/slug both sides
   consistently with how cards render their title), else null. **PURE, TOTAL, no fs** — the purest-
   module discipline of `detectCollisions`.
2. **`proposeEpicEffect` adopts before minting.** Gather the live epics' **titles** alongside their
   ids (extend the `listIdsIn` read, or add a sibling that reads each `E-*.md` frontmatter title), then
   **before** `nextEpicId`: if `findExistingByTitle(card.title, liveEpics)` returns an id, **adopt** it
   — return `{ ok: true, detail: "idempotent — '<title>' already minted as <id>", produced: <its
   path>, artifacts: [<its path>] }` — minting nothing. Only when the title is new does it mint
   `nextEpicId` as today (back-compat: a genuinely new proposal still mints max+1, writes, returns).

**Recommended: adopt, not refuse** — adopting lets a retried chain proceed on the existing card (still
clears); refusing (a `duplicate-title` andon) would leave the card undecomposed and clear 0. **Record
the call** in the work artifact (adopt-idempotent recommended; refuse-andon the alternative).

**Honest boundary:** this makes the EFFECT idempotent on title; it does **not** change *why* the
effect re-ran (the cast/chain retry — a separate, lower-priority observation). `detectCollisions` /
E-004 and the decompose-side `IdCollisionError` are **unchanged** (id-reuse is a different failure).
