## Context

The pure decision + reporting of the history audit — no git, no worktrees, no spawning (T-034-02 does
the impure sweep). Mirror the `head-build-core.ts` split: the impure runner gathers per-commit results,
this core *classifies + renders* them. Create `src/ci/history-core.ts` with two pure functions:

1. **`classifyHistory(results)` — the verdict + report.** Given the per-commit outcomes — model each as
   `{ sha: string; subject: string; green: boolean; summary?: string }` (the impure sweep fills these
   from the worktree build, reusing `classifyBuildOutcome`) — return `{ anyRed: boolean; redCount:
   number; report: string }`. The report lists each red commit (`sha subject` + the failure summary,
   the E-008 "name the failure" style) and a one-line tally; **honest-empty** — an empty `results`
   array renders a clear "no commits in range" line, not an error or a misleading "all green". The
   caller exits non-zero iff `anyRed` (so CI fails on a red commit in range).
2. **`boundRange(allShas, opts)` — the no-silent-cap bounder.** Given the full resolved commit list (the
   impure caller got from `git rev-list`) and a bound (`max`/default), return `{ covered: string[];
   droppedCount: number; note: string }` — the commits to actually sweep + a **loud** note stating
   `covered N of M (bounded at K — widen with <range>)` when `droppedCount > 0`. A silent cap that
   audited 20 of 200 commits and read "all green" would be a lie — this makes the bound visible.

Pure/total: no `Bun.spawn`, no `git`, no fs. Keep the shapes exhaustively handled (`tsc` proves the
empty/red/green and bounded/unbounded cases). The impure T-034-02 supplies `results` and the raw sha
list.
