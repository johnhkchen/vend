## Context

Fire the gate automatically — but **not** on E-008's `on-stop` trigger. `on-stop` fires
every Claude-Code response; a full isolated HEAD build there would be **catastrophic
latency** (a ~minute build after every turn). `check:committed` belongs on `on-stop`
(it's a fast `git status`); `check:head` does not.

**Design decision (settled here):** trigger `check:head` at a **coarser boundary** —
the **`on-clear` hook** (pane/ticket boundary, far less frequent) or at lisa-loop
completion. On a broken HEAD, surface a clear andon (block/flag), mirroring E-008.

**Andon / honest scope (R11):**
- **Confirm the hook frequencies first** (`on-stop` = per-turn; `on-clear` = per-ticket-ish)
  — do **not** put a full build on a per-turn hook. Document the chosen trigger + the
  latency rationale.
- **Block-vs-warn** semantics: determine, don't assume (mirror E-008's finding).
- **Fail-open** — a broken or slow gate must never wedge the loop. The hook edit affects
  **future** loops; live trigger verification is deferred.

**Cites:** `.lisa/hooks/` (`on-clear` vs `on-stop`); `.lisa/hooks/on-stop.sh` (E-008 —
the `check:committed` wiring pattern, but the **frequency differs**); the lisa
setup-guide (hook semantics); `T-010-01` (`check:head`).

## Acceptance Criteria

- [ ] `check:head` is wired to a **coarse** trigger (`on-clear` / loop-end, **not**
      `on-stop`), the choice grounded in confirmed hook frequencies + the latency
      rationale, documented in the hook.
- [ ] A broken HEAD surfaces a clear andon (block-vs-warn determined like E-008);
      **fail-open** if the gate can't run.
- [ ] A broken-HEAD demo shows the andon; the per-turn-latency pitfall is documented as
      avoided; the future-loop effect is noted. `bun run check:*` stay green.
