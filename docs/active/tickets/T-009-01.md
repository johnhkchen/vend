## Context

Author ProposeEpic as BAML — **authoring-only** (render via `b.request`, SAP-parse via
`b.parse`; not the transport). `baml_src/propose.baml`: `function ProposeEpic(signal:
string, charter: string, project: string) -> EpicCard`, where `EpicCard` mirrors the
epic card — frontmatter (`id`, `title`, `kind`, `advances[]`, `serves`), the card
stat-block (`manaCost`, `color`, `type`, `rarity`), and the body (`intent`, `value`,
`doneLooksLike`, `context`). Mirror the proven DecomposeEpic BAML pattern.

**Cites:** `baml_src/decompose.baml` + `baml_src/clients.baml` (the `ClaudeStub`
render-only pattern); `src/baml/decompose-bridge.ts` (`extractPromptText`);
`card-model.md` (the `card = {...}` shape); `playbooks/propose-epic.md` (what the play
must output — PE-3/PE-4).

## Acceptance Criteria

- [ ] `baml_src/propose.baml` defines `ProposeEpic(signal, charter, project) ->
      EpicCard` with the card frontmatter + stat-block + body fields; `bun run baml:gen`
      regenerates the client.
- [ ] A unit test feeds a **canned** reply through `b.parse.ProposeEpic` → a typed
      `EpicCard`; a second asserts `b.request.ProposeEpic(...)` renders the inputs into
      the prompt. (No live call.)
- [ ] `bun run check:test` / `check:typecheck` green.
