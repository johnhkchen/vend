## Context

Bring up `/ci` as its **own Dagger module** — a separate program with its own
`package.json`, on the **Node** runtime (Bun-in-containers, not Bun-as-orchestrator),
engine pinned to **`v0.21.4`** in `/ci/dagger.json`, with a thin `/ci/src/index.ts`
router and no gate logic yet. *Dagger invokes; it never defines* — this is only the
shell.

**Guard the boundary hardest:** `/ci` imports **nothing** from the app, and the app
imports nothing from `/ci`. The only contract is the `bun run check:*` command surface.

**Andon (stop, do not improvise):** do **not** run `dagger develop` to regenerate
against a newer engine — pin `v0.21.4`. A version bump is a *reviewed human step*; if
regen seems necessary, surface it rather than bumping (`ci-strategy.md` rule 1). Docker
daemon must be up for `dagger` to provision its engine (~18s cold-start).

**Cites:** `ci-strategy.md` (the `/ci` structure, the Central Rule, pinned `v0.21.4`,
Node runtime); `playbooks/ci-structural-gate.md` (steps 3–5 — one gate = one sub-class
= one file; `index.ts` thin; `/ci` imports nothing).

## Acceptance Criteria

- [ ] `/ci` is its own module: `/ci/dagger.json` (engine pinned `v0.21.4`),
      `/ci/package.json` (Node runtime, the module's own deps), `/ci/src/index.ts` a
      **thin router** with no check logic.
- [ ] `/ci` imports nothing from the app; the app imports nothing from `/ci`.
- [ ] The engine version is pinned and unchanged — `dagger develop` was **not** run as
      a casual step. If a regen blocker is hit, it is surfaced, not bumped.
- [ ] The app's own `bun run check:test` / `check:typecheck` are unaffected.
