## Context

Make the open-model authoring path coherent with E-035's executor selection, prove parse is
provider-agnostic, and record the honest boundary. Three parts:

1. **Render-client follows the executor (coherence with E-035).** Tie the render-client selection
   (T-036-01) to E-035's executor selection: when `VEND_EXECUTOR=openai-compat` (`executorFor`/
   `src/executor/select.ts`), the render targets `OpenModelStub`; default ⇒ `ClaudeStub` (byte-identical
   back-compat). So the rendered request matches the executor's target — the one open-model path:
   **render (BAML, open-model client) → dispense (E-035 `OpenAICompatExecutor`) → parse (BAML)**. Reuse
   E-035's env/selection (don't invent a parallel switch); keep the default path unchanged.
2. **Parse is provider-agnostic (the proof).** Show `b.parse.*` SAP-parses a **canned open-model-style
   reply** into the same typed output as a Claude reply — it parses *text*, not a provider — via the
   subprocess bridge (no live model). This is the half that needs no change; the test makes the
   provider-neutrality explicit.
3. **Record the boundary (no over-claim).** Update `docs/knowledge/stack.md`: open-model support is now
   **config-level at both layers** — authoring (this epic: the `OpenModelStub` client, render + agnostic
   parse) and execution (E-035: the `Executor` + `OpenAICompatExecutor`). And state the **deferred
   remainder** plainly: the **live agentic open-model runtime** (an open model autonomously reading the
   repo, running tools, iterating turns, materializing a ticket — a local agent loop behind E-035's
   `Executor`, or an open-source agent framework) is **not** built; "support" here is the stack
   *targeting* open models, not proven open-model parity on the agentic plays.

Keep it render-only (BAML is not the transport); don't change SAP-parse logic; don't build the agentic
runtime (the named non-goal).
