## Context

**The pure wave-budget core — generalize the sequential spend algebra to a concurrent wave on one
shared wallet.** The wallet (`src/budget/wallet.ts`) + spend-core (`src/engine/spend-core.ts`) are
PURE and assume LINEAR execution: `fitNext` picks one affordable candidate, `debit` subtracts one
cast's actual (summing both denominations), `canAfford` checks one predicted price. A concurrent wave
(`castGraph`'s `Promise.all` over a ready-set) breaks the accounting — each node authorizes against its
own static budget, so parallel casts collectively overspend the macro allocation. This ticket adds the
pure wave algebra; T-048-02 threads it.

**The load-bearing correctness — the two IA-8 denominations DIVERGE under concurrency:**
- **wall-clock** = the wave's **ELAPSED = MAX(branch times)** — overlapping branches cost ~the longer
  one, not their sum. Concurrency *stretches* the wall-clock budget (a real benefit). So for a wave to
  fit the remaining wall-clock, **each** dispatched node's predicted time ≤ remaining time (max-fits).
- **tokens** = **SUM(branch tokens)** — every branch's burn is real (detect-after, IA-8). So a wave
  fits tokens only if the **cumulative** predicted tokens of the dispatched set ≤ remaining tokens.

The existing `debit` (sums BOTH) is correct sequentially but **over-charges wall-clock** for a
concurrent wave — the bug to avoid.

**Two pure functions (a new `wave-budget.ts` beside spend-core, or extend the wallet — house pattern,
type-only imports, no fs/clock/seam):**

1. **`authorizeWave(wallet, readySet, priceOf) → { dispatch, stopped }`** — the `fitNext` generalization
   to a SET. Walk the ready-set in a deterministic order (the `topoSort` tie-break order); greedily add
   a node to `dispatch` iff it still fits: its predicted **time ≤ remaining.timeMs** (wall-clock,
   each-fits) AND the running **cumulative tokens (incl. this node) ≤ remaining.tokens**. Nodes that
   don't fit go to `stopped` (the hard wall at the wave boundary). PURE/TOTAL. (A wave where the head
   fits but a later node's tokens overflow → head dispatched, tail stopped — the correct partial wave.)
2. **`debitWave(wallet, actuals) → DebitResult`** — fold a concurrent wave's SETTLED actuals into the
   one wallet: **tokens SUMMED**, **wall-clock the MAX** of the wave's actual times (the concurrent
   elapsed) — NOT both summed. Floors at 0 + surfaces per-denomination overshoot (reuse the wallet's
   `floorNonNeg`/`overBy`/`DebitResult` shape — IA-8). **A single-element wave MUST equal the existing
   `debit`** (max-of-one = that one; sum-of-one = that one) — back-compat, asserted.

**Unit-tested (no live model):** a wave that all fits; a partial wave (some dispatched, some stopped on
tokens; some stopped on time); `debitWave` summing tokens but taking max wall-clock across a concurrent
wave; the overshoot accounting (tokens overshoot surfaced once, not double-counted); single-node ==
`debit`; the empty wave (no-op).
