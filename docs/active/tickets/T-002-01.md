## Context

Author the `DecomposeEpic` play as BAML — **authoring-only**: render via
`b.request`, SAP-parse via `b.parse`; BAML is **not** the transport (see
`stack.md` and mc-design-eval's `baml_src/clients.baml` `ClaudeStub`, a client
that is never called). The output type mirrors lisa's story/ticket frontmatter so
shapeless work is impossible by construction (poka-yoke). Inputs: the epic, the
charter, and a project-context snapshot. Runs parallel to the S-001 module tickets
(disjoint files: `baml_src/`).

## Acceptance Criteria

- [ ] `baml_src/clients.baml` defines a **render-only** `ClaudeStub` client (never
      called; mirrors the verified pattern).
- [ ] `baml_src/decompose.baml` defines
      `function DecomposeEpic(epic: string, charter: string, project: string) ->
      WorkPlan`, where `WorkPlan` holds ordered `StoryDraft` / `TicketDraft`
      classes carrying lisa frontmatter fields (`id`, `story`, `title`, `type`,
      `status`, `priority`, `phase`, `depends_on`) **plus** per ticket a `purpose`,
      the `advances` it serves, and a `doneSignal` (charter: work judged by
      purpose/value, not formatting).
- [ ] `baml_src/generators.baml` emits the TypeScript client; `bun run baml:gen`
      regenerates `baml_client/`.
- [ ] A unit test feeds a **canned** model reply (no live call) through
      `b.parse.DecomposeEpic` and asserts a typed `WorkPlan`; a second asserts
      `b.request.DecomposeEpic(...)` renders the inputs into the prompt.
