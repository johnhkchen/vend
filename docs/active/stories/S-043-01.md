Part of **epic E-043 (idempotent-mint-guard)** — make `propose-epic`'s mint idempotent so a re-run
can't orphan a card. E-039's live sweep double-minted the doctor epic (E-042 + childless orphan
E-041, same title) because `proposeEpicEffect` re-mints `nextEpicId` = max+1 each run — TOCTOU-safe
against clobbering, but not idempotent under retry. The collision guard (E-004) can't catch it: each
run minted a *fresh* id, never a *reused* one. The stable identity is the **title**.

One surgical ticket: T-043-01 adds a pure title-dedup in `id-guard.ts` (`findExistingByTitle`, beside
`detectCollisions`) and makes `proposeEpicEffect` **adopt** an existing same-title epic before minting
a fresh slot — so a re-run converges on one card and the chain decomposes it (no orphan). Deterministic
proof: run the effect twice with the same card → mints once; distinct titles still mint two. New-title
path + E-004 unchanged. Single-ticket epic (E-027/E-038 precedent). Epic-scoped ids (R6 — `043` clean).

## DAG

```
T-043-01  title-keyed idempotency — pure findExistingByTitle (id-guard.ts) + proposeEpicEffect adopts same-title epic; deterministic twice-run proof; new-title path + E-004 unchanged
```

**Out of this slice:** changing the cast/chain retry mechanism (why the effect re-ran — separate,
lower priority); refusing genuinely-new proposals; touching `detectCollisions`/E-004 (id-reuse, a
different failure); a content-hash idempotency key (title suffices for the observed incident).
