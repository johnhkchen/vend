**Scope:** the turn-cap seam between the cast loop and the executor — `decomposeEpicPlay.maxTurns` (src/play/decompose-epic-core.ts, `DECOMPOSE_MAX_TURNS` = 15) is passed to the Claude executor as `--max-turns` (src/executor/claude.ts `buildArgs`), while the run record's `turnsUsed` comes back from the executor result's `num_turns`. Three successful live decompose runs this week logged **16, 17, and 18 turns against the 15 cap** (`turns: 18 / 15 cap`, run-2026-07-13T02-00-52) — so either the cap is not binding, or the two counters measure different things (e.g. `--max-turns` bounds assistant turns while `num_turns` counts more) and the `N / cap` display compares unlike units. Diagnose which, then make the surface honest.

**Story acceptance:** the mismatch is root-caused and pinned in a test (what `--max-turns` bounds vs what `num_turns` counts, with the evidence cited in the work dir); after the fix, a successful run can never display `turns: N / cap` with N > cap in same-unit terms — either the enforced cap is real and the displayed count is the capped quantity, or the display names two honestly-labeled numbers; existing DECOMPOSE_MAX_TURNS pin tests updated only if the documented judgment itself changes.

**Honest boundary:** FREE to diagnose (ledger + docs + code reading; no metered cast required — three live evidence runs already exist in `.vend/runs.jsonl`). If the root cause turns out to live inside the Claude CLI's own flag semantics, the honest fix is display/labeling + a pinned characterization test, not a re-implementation of turn counting. Ceiling: decompose's cap seam only; do not generalize to a cross-play turn-policy framework.

**Out of this slice:** changing DECOMPOSE_MAX_TURNS's documented value (15 stays unless the diagnosis proves the judgment itself was measured in the wrong unit); turn caps for other plays; any retry/escalation policy on cap-hit.
