## Intent — the bigger-picture play

The project has two structural gates, and a hole between them. **`check:committed`** (E-008, the
`on-stop` hook) enforces *that source is committed* before a session ends. **`check:head`** (E-010, the
`on-clear` hook) verifies *HEAD builds in an isolated worktree* when a **ticket clears**. Neither
enforces **test-green on each commit** — so a mid-ticket red commit survives in history even after the
ticket goes green (T-031-01's `3dfb95f` shipped a failing test; it was caught *only* because the loop
left the follow-up fix uncommitted and `check:committed` tripped on the dangling source — pure luck).

This builds the missing layer: a **git `pre-commit` hook** that runs `check:test` and **blocks a red
commit before it enters history** — naming the failing tests, the E-008 "fail naming the file" style.
The git pre-commit hook is the right seam precisely because the loop's agents commit via plain
`git commit`; the hook fires on every one of them (lisa has no `on-commit`). The three layers then
compose into a complete story: **per-commit** (tests green) · **per-stop** (source committed) ·
**per-clear** (HEAD builds).

Two disciplines carried from the existing gates: the hook **only invokes** — the policy lives in a pure
`src/ci/precommit-core.ts` (testable), never in the shell (the `ci-strategy.md` central rule); and it
**fails open on inability to run** (no bun, not a repo, checker error ⇒ allow the commit — a broken
checker must never wedge the loop) while **failing closed on a genuine red** (tests ran and failed ⇒
block). And because a gate that can be silently un-installed is no gate, the hook ships with an
installer **and** a `check:hooks` guard that flags when it isn't active (the E-012 "police the gate"
move).

## Value to the design

Closes the gate hole and makes the autonomous loop's history **honestly green per commit** — P3
(consistency / the gate-as-contract: a commit is a valid unit, `git bisect` stays meaningful) and P4
(loop integrity: the loop can't silently commit red, even mid-ticket). It is the third corner of the
E-008/E-010 gate frame — the gates that make "done" mean *committed, builds, and green*.

## Done looks like

- **A git `pre-commit` hook** (committed, version-controlled via `core.hooksPath` → a `.githooks/` dir)
  that runs `check:test` before each commit: **green ⇒ allow; red ⇒ block** (non-zero, naming the
  failing tests); **can't-run ⇒ allow** (fail-open). A thin invoker — policy lives in the core.
- **A pure `src/ci/precommit-core.ts`** — `classifyPrecommit(runResult)` deciding block / allow + reason
  (`green` | `tests-failed` | `could-not-run`) + the andon message; unit-tested (the E-008
  `committed-core.ts` precedent).
- **An installer + a `check:hooks` guard** — `hooks:install` sets `core.hooksPath`; `check:hooks`
  reports when the hook isn't active so the gate can't be silently absent (E-012 spirit). Activated in
  this repo.
- **Proven free + deterministic:** a red working tree → the hook **blocks** the commit naming the
  failure; a green tree → the commit proceeds; `check:hooks` reports active — all without an LLM cast.
  `bun run check:*` green.

## Context & constraints

- **Cites:** `.lisa/hooks/on-stop.sh` (the E-008 invoker to mirror — "only invokes", fail-open, reads
  `check:committed`); `src/ci/check-committed.ts` + `src/ci/committed-core.ts` (the invoker/pure-core
  split + the `*-core.test.ts` test precedent); `src/ci/check-head.ts` (the E-010 sibling gate);
  `package.json` (the `check:*` script convention — add `check:hooks` + `hooks:install`; `check:test` =
  `bun test`); `docs/active/epic/E-008.md` / `E-010.md` / `E-012.md` (the gate-tightening lineage);
  `docs/knowledge/ci-strategy.md` (the "hook only invokes; definition lives in core" central rule).
- **Scope of what the hook runs:** `check:test` (the per-commit test-green headline — the T-031-01
  miss); include `check:typecheck` too if it stays sub-second (compiles + green). **NOT** `baml:gen`
  (slow; codegen rarely changes per commit and `check:head` runs the full `check` at clear). Keep it
  fast — the loop commits several times per ticket.
- **Fail-open vs fail-closed (the load-bearing nuance):** *inability to run* the check (no bun, not a
  repo, spawn error) ⇒ **allow** (fail-open, mirroring on-stop — never wedge the loop); the check *ran
  and tests failed* ⇒ **block** (fail-closed — the gate's whole job). `classifyPrecommit` distinguishes
  these; both are unit-tested. (Git's `--no-verify` remains the human escape valve; the loop doesn't use it.)
- **Non-goals:** a retroactive history sweep / `git bisect`-for-red-commits backstop (a downstream
  pull — prevention first, post-hoc audit later); running the *full* `check` (with `baml:gen`) per
  commit (too slow; `check:head` owns the full build at clear); a lisa `on-commit` hook (lisa has none —
  the git pre-commit hook is the seam); changing `check:committed`/`check:head` (this is the third
  layer beside them, not a change to them). Not an executor (**N4**).
- **First slice = the pure policy core + the pre-commit hook + the installer + the `check:hooks`
  guard, activated.** **Out:** the retroactive bisect sweep, full-check-per-commit.
