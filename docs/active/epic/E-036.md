## Intent — the bigger-picture play

E-035 made the **execution seam** pluggable (the `Executor` interface + a second, OpenAI-compatible
executor). But Vend's stack has a second LLM touchpoint: **BAML, the authoring layer**. BAML is
render-only here (`clients.baml`: *"Vend uses BAML for AUTHORING ONLY — `b.request.*` renders the prompt,
`b.parse.*` SAP-parses the reply; the dispense rides the seam, NOT BAML's transport"*). The one BAML
client today is `ClaudeStub` (provider `anthropic`) — it shapes the request, and is never called.

This epic adds the **open-model half** of authoring: declare an **open-model BAML client**
(`openai-generic` — the universal OpenAI-compatible provider, the same shape E-035's `OpenAICompatExecutor`
targets) and show the authoring layer renders against it (request-shaping in the open-model provider
format) while SAP-parse stays **provider-agnostic** (it parses reply text, not a provider). Paired with
E-035, the full open-model path is then capable **at config level**: render the request for the open
model (BAML) → dispense to it (E-035's executor) → parse its reply (BAML, provider-agnostic).

**The honest boundary — what "support" means, and what it does NOT (load-bearing, the user's own scope).**
"Showing support via BAML suffices" — so this delivers the **authoring-layer capability**: an open-model
client exists, request-shaping renders for it, parse is provider-neutral. It does **not** build the
**agentic open-model runtime** — the runner that has an open model autonomously read the repo, run
tools, iterate turns, and materialize a ticket. That runtime (a local agent loop behind E-035's
`Executor`, or an open-source agent framework) is the **deferred remainder** of the agentic-runner
thread, named here, not claimed. We show the stack *supports* open models at both layers; we do not
claim an open model can yet autonomously clear work.

## Value to the design

Completes the **config-level** open-model story (P6): with E-035 (execution) + E-036 (authoring), every
LLM touchpoint in Vend can target an open model — the clearest evidence yet that Vend is model-plural by
design, not Claude-only. The remaining gap (the live agentic runtime) is isolated and named, so the
claim stays exactly as big as the evidence.

## Done looks like

- **An open-model BAML client** — `OpenModelStub` (provider `openai-generic`, `base_url`/`model` from
  env, **render-only, no secret sent**) in `clients.baml`, mirroring `ClaudeStub`'s render-only pattern
  + guard. `baml:gen` green with it declared.
- **Request-shaping targets it** — a function rendered against `OpenModelStub` builds a request in the
  **openai-generic provider format** (distinct from `ClaudeStub`'s anthropic format — a real, assertable
  difference), selectable at render time (BAML's runtime client selection).
- **Parse stays provider-agnostic** — `b.parse.*` SAP-parses an open-model-style reply the same as a
  Claude one (it parses text, not a provider).
- **Coherent with E-035** — render-client selection follows the executor selection (`VEND_EXECUTOR=
  openai-compat` ⇒ render via `OpenModelStub`), so the rendered request matches the executor's target:
  render → dispense → parse, one open-model path.
- **The boundary recorded** — the authoring+execution open-model support is documented as config-level;
  the live agentic runtime is named as the deferred remainder (a knowledge-doc note), not over-claimed.
- **Proven free + deterministic:** `baml:gen` green; render-against-`OpenModelStub` yields an
  openai-format request (vs ClaudeStub's anthropic format); `b.parse.*` parses a canned open-model reply
  — all via the existing subprocess-bridge, **no live model**. `bun run check:*` green.

## Context & constraints

- **Cites:** `baml_src/clients.baml` (`ClaudeStub` — the render-only client pattern + the "never called"
  comment/guard to mirror); `baml_src/generators.baml` (the codegen); `src/baml/decompose-bridge.ts`
  (`extractPromptText`/`runOp` — `b.request.*`/`b.parse.*`, the render/parse bridge + the
  one-call-per-process subprocess pattern); the generated `baml_client/` (BAML's **runtime client
  selection** — `ClientRegistry` / a `b.request` client override; **verify the exact API in the
  generated client** before wiring, the one external contract); `src/executor/select.ts` (`executorFor`/
  `VEND_EXECUTOR` — E-035, the executor selection to follow); `docs/knowledge/stack.md` (the
  authoring-only-BAML + open-model intent — the doc to update).
- **Render-only, no secrets (the `ClaudeStub` discipline):** `OpenModelStub` is shaped for `.request`
  and never dispatched from BAML; `base_url`/`model` from env (`VEND_OPENAI_BASE_URL`/
  `VEND_EXECUTOR_MODEL`, the E-035 names), a dummy/optional key never sent. Local-first.
- **The demonstrable difference is the request FORMAT, not the prompt text** — `openai-generic` shapes
  the request body (messages/model/url) in OpenAI format vs `ClaudeStub`'s anthropic format; assert on
  that (the prompt *text* may be identical — don't contrive a text difference).
- **BAML one-call-per-process limit:** any render/parse proof runs through the existing subprocess
  bridge (the `decompose-bridge.ts` pattern), not inline in `bun test`.
- **Non-goals:** the **agentic open-model runtime** (the runner that autonomously executes a play on an
  open model — the deferred remainder, the real heavy lift); making BAML the live transport (it stays
  render-only — the dispense rides E-035's `Executor`); a hosted-API client with committed secrets
  (local-first, env-config); per-play client selection (a play declaring its BAML client — scope creep,
  downstream); changing the SAP-parse logic (it's already provider-agnostic). Not over-claiming
  open-model parity (**the boundary above**).
- **First slice = the open-model BAML client + render-targets-it + parse-is-agnostic + the coherent
  E-035 wiring + the recorded boundary.** **Out:** the agentic runtime, per-play client, BAML-as-transport.
