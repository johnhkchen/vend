## Intent — the bigger-picture play

E-047 cast the survey → [propose-epic ×2] → capture-note diamond LIVE and proved concurrency (two real propose casts overlapped ~69s), but the 2-upstream JOIN never ran: each node drew on its OWN per-node budget, propose-1 hit budget-exhausted, castGraph correctly halted, and capture-note was skipped — the join stayed stub-proven only. E-048 then gave castGraph an optional ONE shared Wallet across the wave (authorizeWave/debitWave), but that envelope was never threaded through the real-play harness: castRealPlayGraph still calls castGraph(nodes, edges) with no wallet (src/play/graph-real-play.ts:168), budgeting node-by-node. This epic threads E-048's shared wallet through castRealPlayGraph — allocate ONE envelope once so BOTH proposes draw from it and both finish — then re-casts the diamond LIVE and proves the join: capture-note RUNS and receives BOTH proposes' produced (the 2-entry NodeUpstreams map that runGraph/castGraph express but runChain structurally cannot). It matters NOW because this is the single remaining stub-only property on the E-046/E-047/E-048 substrate, and E-050 (self-funding) + E-051 (no AskUserQuestion hang) just made the unattended metered cast robust enough to run it for the first time.

## Value to the design

Realizes the live, end-to-end proof of the multi-upstream JOIN — the one DAG capability that runChain cannot structurally express. It directly advances P7 (the shared wallet bounds the whole wave with one hard envelope, so budget no longer leaks per-branch as it did in E-047), P4 (the join runs autonomously to a settled verdict with no human in the loop), and P6 (the proof exercises the executor-agnostic castGraph seam, not a Claude-Code-specific path). With this, every property of the substrate — fan-out, wave dispatch, halt-dependent-subgraph, shared-wallet semantics, AND the live join — is either deterministically or live-proven, with no stubs left.
