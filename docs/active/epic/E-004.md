## Intent — the bigger-picture play

`DecomposeEpic` currently mints story/ticket ids without checking the target board;
the E-001 live proof showed it reusing `S-001`/`T-001-01…` (safe only because A1
materialized into a sandbox). Add a cross-board uniqueness guard — refuse-on-collision
or an epic-scoped id namespace — so a run against a populated board cannot silently
overwrite existing work.

## Value to the design

A gate that prevents a destructive write (**P3**). It is the prerequisite to
machine-decomposing any further epic *in place* rather than by hand or in a sandbox —
the highest-leverage kaizen surfaced by E-001.

## Done looks like

- Materialize **refuses** (andon, no write) when any generated id collides with an
  existing id on the target board — or ids are namespaced by epic so collision is
  impossible.
- A run against a populated board writes only fresh ids, or stops with a named
  reason — never silently overwrites.
- A colliding-fixture test proves the refusal.

## Context & constraints

- Source: `.vend/runs.jsonl` A1 / `docs/active/work/T-002-04/proof.md` kaizen #4.
- Touches `src/play/materialize.ts` (the write) and `project-context.ts` (it already
  lists existing ids — feed them to the guard).
- Out of scope: a full id-allocation scheme; the minimal guard is refuse-on-collision.
