## Intent — the bigger-picture play

E-039's macro-wallet sweep cleared `vend doctor` — and **minted it twice**: E-042 (decomposed, logged)
and **E-041** (a childless duplicate, same title `vend-doctor-preflight`, never logged). The loop
adjudicated and deleted the orphan, and filed this fix. The root cause is precise:

- `proposeEpicEffect` (`src/play/propose-effect.ts:65`) **re-mints** the id via `nextEpicId(live)` =
  **max existing + 1** against the live board (design D2: TOCTOU-safe — it can never *clobber* a
  hand-authored id). But that same policy makes it **non-idempotent under re-invocation**: run the
  effect twice for one proposal and each call grabs the *next* free slot —
  - run 1: board …E-040 → mint **E-041**, write the card, then (a timeout/transient) before logging;
  - run 2 (retry): board now has E-041 → mint **E-042**, write the *same title*, log + decompose.
- `detectCollisions` (`id-guard.ts`, the E-004 guard) passes both times — each minted a *fresh* max+1
  id, never a *reused* one. **The collision guard catches id REUSE; it cannot catch a fresh id for a
  duplicate PROPOSAL.** The stable identity of "the same proposal" is its **title** (the model's
  `card.id` is ignored/re-minted; the title is what's stable across a retry).

**The fix — title-keyed idempotency.** Before minting, check whether an epic with this **title**
already exists on the board. If so, **adopt it** (return ok, `produced` = the existing card's path,
`detail: "idempotent — '<title>' already minted as E-0XX"`) instead of grabbing a fresh slot. A
re-run then **converges on the one card** and the chain decomposes it — no orphan. Mirror the E-004
shape: a pure title-dedup in `id-guard.ts`, composed in the effect, which now reads the live epics'
titles (a small impure gather) alongside their ids.

## The honest boundary — what this fixes and what it does NOT

- **It makes the propose EFFECT idempotent on title; it does NOT change WHY the effect re-ran.** The
  retry/re-invocation lives in the cast/chain layer; this fix makes the effect *safe to run more than
  once* regardless of the cause (the right layering — the verdict asked for the guard "in
  propose-epic"). Naming the re-invocation source is a separate, lower-priority observation.
- **Title is the idempotency key** — sound because the model produces a stable title on a retry of the
  same inputs, and two epics with the *same* title are a smell either way (a retry to dedupe, or a
  genuine duplicate to refuse). A wildly-different title on retry would still double-mint (the
  uncommon edge; the common retry case is covered).
- **Adopt, not refuse (recommended).** Adopting lets the chain proceed on the existing card (the
  retry still *clears*); refusing (an andon) would leave the first card undecomposed and clear 0.
  **Open sub-question** (record the call): adopt-idempotent (recommended) vs refuse-andon
  (`duplicate-title`) — recommend adopt, so a retried sweep converges and clears.
- **Decompose's own id-collision guard (E-004) is unchanged** — it still protects the tickets a minted
  epic decomposes into.

## Value to the design

Closes a correctness gap the autonomous loop **demonstrably hits** (we watched it). An unattended
macro-wallet sweep that retries must not litter the board with orphan cards — P3 (the board is the
contract) and P4 (autonomy you can leave alone). Cheap, surgical, grounded in a real incident.

## Done looks like

- **A pure title-dedup** in `src/play/id-guard.ts` (beside `detectCollisions`) —
  `findExistingByTitle(cardTitle, existing: {id,title}[]) → existingId | null` (or equivalent): PURE,
  TOTAL, deduped, no fs. Slug/normalize titles consistently with how cards are rendered.
- **`proposeEpicEffect` is idempotent on title** — it gathers the live epics' titles (alongside
  `listIdsIn`), and if `card.title` already exists, **adopts** the existing card (ok, `produced` =
  its path, an "idempotent — already minted as E-0XX" detail) instead of minting a fresh `nextEpicId`.
  The mint path is unchanged when the title is new (back-compat — a genuinely new proposal still
  mints max+1).
- **Deterministic proof (free):** a test that `proposeEpicEffect` run **twice** with the same card
  against a temp board mints **once** (the second run adopts the first — no E-04x orphan), and that
  two **distinct**-title cards still mint two epics. Map it to the E-041 incident (the second run
  would have adopted E-041, no E-042 orphan). `detectCollisions`/E-004 behavior unchanged. `bun run
  check:*` green.
