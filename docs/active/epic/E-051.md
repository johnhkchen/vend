_Proposed by Vend's `propose-epic` play._

## Intent — the bigger-picture play

Autonomous casts run headless via `claude -p` (src/executor/claude.ts buildArgs/dispense), but the piped agent can still invoke Claude Code's built-in AskUserQuestion tool — and no answerer exists on a piped cast, so the call hangs the cast until it resolves empty or the wall-clock guard kills the pane. E-049's decompose hit this live: the agent improvised a mid-decompose question and stalled. This is the reliability gap the dogfood surfaced and the last thing keeping `vend chain`'s decompose from being fully unattended. The play: make interactive tools UNAVAILABLE on the autonomous-cast path so the agent proceeds autonomously — a clean unavailability, never a hang. Thread a `disallowedTools` option through buildArgs to a `claude -p --disallowedTools` flag covering AskUserQuestion, the symmetric denylist beside the existing `--allowedTools` allowlist (claude.ts:164). Scope is the autonomous plays only (propose-epic, decompose-epic, work); interactive/TUI surfaces are untouched.

## Value to the design

Realizes autonomy-by-default (P4) as an enforced property of the headless executor rather than a hope: the unattended loop cannot be stalled by a prompt it has no way to answer. It hardens the clearing chain's most autonomous moment (decompose) and removes the last manual-babysit failure mode the dogfood exposed, directly advancing the core feature — work that is safe to run autonomously.

## Done looks like

An autonomous cast (propose-epic / decompose-epic / work) emits a `--disallowedTools` flag denying AskUserQuestion, and an agent that attempts the tool mid-cast finds it unavailable and proceeds rather than hanging; a decompose that previously stalled now completes unattended. Interactive/TUI casts emit no such flag and are byte-for-byte unchanged. buildArgs is asserted PURE: the disallow flag is present for an autonomous cast and the no-tools argv is otherwise identical — no live model needed to prove it.

## Context & constraints

Ground against src/executor/claude.ts: buildArgs is a PURE argv builder that already threads `allowedTools → --allowedTools` (E-032, claude.ts:131-164) with the back-compat rule that an empty/omitted list emits no flag (argv byte-identical to pre-flag). Add `disallowedTools` as the symmetric option beside it, following the same empty-omitted-emits-nothing discipline so non-autonomous paths are provably unchanged. PREREQUISITE: verify the exact flag spelling and value format against `claude -p --help` before wiring (as E-032 did for `--allowedTools`) — the card assumes `--disallowedTools` but the build must confirm. Resolved tool-set values are threaded in at cast (cf. T-032-02); this epic supplies the denylist plumbing and routes it only through the autonomous-cast options, not the interactive surface. NON-GOALS respected: this is not a supervision/approval prompt (N2) — the fix removes the human-in-loop dependency rather than adding a babysit gate; it stays Claude-Code-specific tool plumbing behind the existing executor interface and does not leak into the executor-agnostic contract. OUT OF SCOPE: building an in-cast answerer for AskUserQuestion, broader tool-policy redesign, or changing TUI/interactive behavior. Right-sized as one ~2-hour feature block (pure-function extension + autonomous-path threading + deterministic tests).
