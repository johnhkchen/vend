_Proposed by Vend's `propose-epic` play._

## Intent — the bigger-picture play

Add a THIRD consumer of the existing E-021 Projection IR — a pure `projectionToSvg(projection, overlays?) -> string` that renders the frozen Projection to one large static SVG, plus a thin CLI/file seam to write the `.svg`. Today `project.ts` projects the graph's tickets into a deeply-frozen Projection and `paper.ts` renders that same IR to markdown/Mermaid (its own header names itself the 'second consumer' and invites a third). This epic delivers the visual surface the Linear renderer was meant to provide, but as a file rather than a live MCP integration — so a designer/founder sees the whole board at a glance without the Linear-MCP blocker. It matters NOW because Frontier 4 (the non-dev round-trip) is otherwise stalled on that blocker; the SVG renderer is the unblocked path to the same value. Render each ProjectionGroup (grouped by `spec.groupBy`) as a labeled swimlane/column, each ProjectedCard as a box carrying its Card face text + `color` token, and each ProjectionLink ('depends_on') as an edge between two card boxes, using a deterministic hand-rolled grid/swimlane layout — no force-directed or external layout engine.

## Value to the design

Realizes a second visual rendering capability over the multi-consumer Projection contract, proving the E-021 data/presentation split was worth paying for (a new surface costs a renderer, not a re-projection). It advances P5 (local-first): the output is a portable file usable fully offline with no external service. It advances P3 (gates are the contract): the renderer's quality is an enforceable, deterministic gate — pure (no clock/random), one `<rect>` per card, one group label per group, one edge per link, honest-empty yields a valid minimal SVG, so the same projection yields byte-identical output and correctness is checked rather than asserted.

## Done looks like

A pure `projectionToSvg(projection, overlays?) -> string` exists in `src/present/`, mirroring paper.ts's contract exactly (frozen Projection + optional overlays in, honest-empty default, never writes back the projection, MCP-independent, no live model). Running the file-output seam on the live board writes a valid `.svg` that opens in any viewer and shows every group as a labeled swimlane, every card as a labeled box carrying its color token, and every depends_on link as an edge. A deterministic test suite (no live model) passes: one rect per card, one group label per group, one edge per link, honest-empty renders a valid minimal SVG, and rendering the same projection twice is byte-identical.

## Context & constraints

Ground against `src/present/`: `project.ts` (the Projection IR — `Projection { groupBy, colorLanguage, groups: ProjectionGroup[], links: ProjectionLink[] }`, with ProjectedCard/ProjectionGroup/ProjectionLink/ProjectionOverlays, deeply frozen) and `paper.ts` (the existing second consumer; mirror its (Projection, overlays?) signature, its honest-empty discipline, and its purity exactly). PREREQUISITES, already satisfied — do NOT rebuild them: the Projection IR and `projectGraph` (E-021), the `authority-guard.ts` / one-way-authority discipline to reuse, and the `spec.groupBy`/`colorLanguage` knobs. Layout must be deterministic and dependency-light (hand-rolled SVG in the house pure-render style) — explicitly NOT a force-directed or external layout engine. Respects non-goals: N4 (pure renderer, executes nothing), N2 (a static glance-able file, not a live babysitting/step-approval dashboard), N3 (a reusable renderer over the IR, not a one-off prompt). OUT OF SCOPE and named as the follow-on, not hidden here: the Linear-MCP live integration (still deferred/blocked) and the annotation→demand round-trip (what teammates do with the rendered board). Decomposes ~like paper.ts's E-021 slice — the pure SVG renderer core + the CLI/file seam — which is one ~2-hour feature block; it does not need to be split into several epics.
