## Intent — the bigger-picture play

Run a soft-launch dress rehearsal of the kitchen QuickStart seed on EmDash + Astro 6 to harden the cold-start path — the goal is a RELIABLE BOOTSTRAP, not feature depth. Stand up the minimal clone-and-drive seed (a `Dish` content type the cook fills via the EmDash admin, plus a deliberately-stubbed storefront at the site root) and drive it once: brew-installed vend, `vend init`, a green `vend doctor`, then `vend steer` yields a coherent board and `vend work` CLEARS the single menu-render slice — read Dish content from EmDash's REST API and render a simple mobile-first menu, one card per dish (photo, name, description). The real purpose is to shake out and FIX every bootstrap friction surfaced along the way — install, init, doctor, the template scaffold, seed-intent reaching steer, the cold-start budget, and the codebase-memory-mcp absence — and to capture a gold-master EXPECTED-OUTCOME.md from this run. This matters NOW because it is the hardening phase that earns the right to the real proof: a fresh, untouched repo we drive once as the forward-E1 autonomy witness. It reuses already-shipped seams (kitchen `--template` E-058, seed-intent-reaches-steer E-059, fresh-seed graceful-degrade + cold-start budget E-060, brew-installable vend E-061) rather than building new engine surface.

## Value to the design

Realizes a trustworthy onboardable cold-start: the four shipped distribution/seed seams are exercised together for the first time under a real-cook scenario, every friction point gets fixed at source, and the run is frozen into a re-runnable EXPECTED-OUTCOME.md gold-master. This advances the core feature (clearing intent into allocatable work) by making the FIRST clearing a non-expert can reach reliable — and it advances P2 (the two-gesture transaction becomes genuinely onboardable), P5 (the whole path runs local-first on one machine), and P4 (witnessing autonomy clear one real menu-render slice against its gates on a budget).

## Done looks like

On a machine with brew-installed vend, the seeded EmDash+Astro kitchen repo drives end-to-end in one pass: `vend init --template kitchen` scaffolds the Dish type + stubbed storefront, `vend doctor` is green, `vend steer` produces a coherent kitchen board, and `vend work` clears the menu-render slice so the storefront renders a mobile-first menu (one card per dish, photo/name/description) from EmDash's REST API. Every bootstrap friction hit during the rehearsal is fixed (not annotated), and a gold-master EXPECTED-OUTCOME.md is committed capturing this run as the consistency bar for the later clean-room drive.
