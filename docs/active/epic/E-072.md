_Proposed by Vend's `propose-epic` play._

## Intent — the bigger-picture play

The 2026-07-12 UX survey (docs/active/pm/ux-rubric-survey.md) graded vend's operator surface across nine dimensions; three came back 2/5 — the floor of the board — and all three are the same failure in three places: the CLI doesn't meet the operator where they are. (1) The most-typed gesture in any CLI, `vend --help` / `vend help`, is a dead end that errors, the usage block hides five real commands (doctor, user-guide, envelope, audit, --version) so the whole trust/audit surface is undiscoverable, and a wrong argument throws a 9-line usage wall instead of naming the one thing that's wrong. (2) During a metered cast the operator sees ~90 bare event names (`· system (thinking_tokens)`) and cannot tell how far along, how much spent, or whether it's alive — the literal UX face of the dogfood's 'not hands-off' verdict; the walk-away promise is invisible exactly while it's being kept. (3) The budget flag speaks machine units — the operator types milliseconds and raw token counts at the counter, while the shelf already speaks `~40m/400k`. This is polish, but it is load-bearing polish: it's the difference between a transaction a person trusts and one they babysit. Worth clearing now because the surface is otherwise mature (v0.4.0-rc era, six dimensions at 4–5) and these three are the cheap, high-leverage lifts that make the rest legible.

## Value to the design

Realizes discoverability, live legibility, and humane input at the CLI counter. Advances P2 (the two-gesture transaction stops requiring undocumented-command recall or millisecond arithmetic), P4 (a visible elapsed/spend/turn line is what makes walking away a rational choice rather than a leap of faith — supervision replaced by trustworthy progress, not a dashboard), and P7 (the budget contract becomes legible at the moment of spending and at the moment of funding, in the same units both ways). It hardens the core clearing guarantee at its most-touched seam: the surface where a human actually picks, funds, and lets go.

## Done looks like

An operator who has never read the code can, from the terminal alone: run `vend --help` (or `vend help`) and get a complete usage answer listing every real command grouped free-vs-metered; mistype a command and be told the one wrong thing with a targeted fix (`did you mean steer?`) instead of a usage wall; watch a metered cast and see one updating line of elapsed time, price-true weighted spend against the funded envelope, and turn count in place of the raw event stream; and fund a run with `--budget 40m,350k` (with `<ms>,<tokens>` still accepted) and see the parse echoed back in the same humane form the shelf speaks. The three survey dimensions (2/4/8) re-grade above their 2/5 floor.

## Context & constraints

Grounds against the surface graded in docs/active/pm/ux-rubric-survey.md at HEAD (post E-071, v0.4.0-rc.2). The three slices are the three 2-of-5 dimensions: discoverability (§2), live feedback (§4), humane units (§8). PE-7 note: this is at the upper edge of one 2-hour feature block — three genuinely separable slices. Slice 2 (the live progress line: price-true weighted spend accumulation, envelope math, per-turn update against streamed executor events) is the heaviest and, if decompose finds it overflows, is the natural cleave point into a second epic; slices 1 and 3 are lighter and cohere as string/parse/usage work. DecomposeEpic should size this and split if slice 2 dominates. Prerequisites (name, don't hide): the price-true weighting for spend already exists in the budget/ledger path (the envelope already renders `~40m/400k` on the shelf, so the humane formatter and the weighting both exist to reuse — this is surfacing, not inventing); the executor already emits streamed turn/token events (slice 2 reads them, doesn't add them). Out of scope and staged elsewhere per the signal: the SVG/vocabulary work (dimension 9), detached/notify (its own design pull), and any TUI beyond the existing line-oriented stdout — this epic stays inside the current print-a-line surface. Respects N1/N2 (a progress line is not a babysitting dashboard and adds no approval step — it makes unsupervised running legible, the opposite of supervision) and N3 (touches the CLI gesture surface, not the playbook unit).
