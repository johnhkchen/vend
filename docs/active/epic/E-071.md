## Intent — the bigger-picture play

The routing arc (E-069/E-070) makes seat choice a STATIC decision: the operator either pins `--agent codex` or the board mints unrouted on lisa's default seat. That treats the two seats as interchangeable name-tags, but they aren't — each seat draws on its own quota that refills on its own reset window, and in real two-seat operation one lane runs hot (near/at its cap, throwing 429s) while the other sits cool. Today the operator notices this by hand and pins the cooler seat manually — supervision-after-the-fact, the exact babysit E-069 set out to remove, just moved one rung up. This epic makes the DEFAULT seat overflow-aware: when a board-writing gesture omits `--agent`, Vend reads recent lane heat from the run ledger (usage burned against each seat's reset-window quota, plus any 429/cap signal on that lane) and stamps the cooler seat as the inferred default. An explicit `--agent` still wins — operator override is a pin, not a suggestion. This is the dynamic next rung of the ratified two-seat operator's ladder: it turns 'pick a seat' into real capacity scheduling and is the point where lane-denominated budget — quota-per-reset-window as a first-class denomination — becomes the thing Vend actually allocates against. It matters NOW because routing usage across the claude and codex lanes is already routine operator work, and a capped lane silently stalling casts on the hot seat is the failure the static arc can't see.

## Value to the design

Realizes the core allocation feature made live: the clearing house schedules work across independently-metered capacity, not just onto a named executor. It advances P7 by promoting quota-per-reset-window to a budget denomination Vend honors both ways — a hard contract now reads at the lane level, so casts don't pile onto an exhausted lane. P4: the cooler-seat decision is made by the system from the ledger, not by an operator watching for 429s, so overflow handling stops being supervision. P2: the seat drops out of the counter gesture entirely in the common case — the operator picks + budgets + goes, and the right lane is inferred. P6: lanes are just seats behind KNOWN_SEATS, so the routing stays executor-agnostic — the board carries the scheduling decision, no gesture assumes a specific executor. It strengthens clearing itself: a cleared unit settles onto the lane that actually has capacity to run it.

## Done looks like

On a project whose ledger shows one lane recently hot (near/at its reset-window quota or having thrown a cap/429 signal), a board-writing gesture with NO `--agent` flag stamps `agent:` = the cooler seat on every materialized ticket, automatically. When both lanes read cool, the mint stays unrouted and is byte-identical to today's default (no `agent:` key). An explicit `--agent <seat>` always overrides the inference (operator pin wins). The run record carries an honest, countable marker of the inferred lane heat and which seat was chosen and why — the overflow-routing analog of the seatDefaulted marker — so requested-vs-actual and inference-vs-default stay auditable in provenance. Observably: a capped lane's next casts land on the other seat with no hand-editing and no live babysit.

## Context & constraints

Ground against the shipped routing arc — `src/play/agent-seat.ts` (KNOWN_SEATS, the seat guard), the materialize effect's seat stamping in `decompose-effect.ts`, and E-070's seatDefaulted one-way marker in `src/log/run-log.ts` — plus the run ledger this epic must read (`src/log/run-log.ts` RunRecord: it records `usage` and `wallMs` per run and the seatDefaulted marker). PREREQUISITE, named not folded in as hidden scope: the ledger does NOT today carry a per-lane heat signal — RunRecord has no seat-of-execution field, no 429/cap marker, and no reset-window quota reference; inferring lane heat requires that substrate (which seat a run burned on, and a cap/429 signal), and if it isn't present it is a prerequisite ticket, not silent scope. Second prerequisite: reset-window quota values and the reset cadence per lane must be SOURCED from the real seat/quota convention, never invented here; KNOWN_SEATS stays the single source of the lane list, reused to schedule, not re-invented. This is RIGHT-SIZED to one ~2-hour block only because it stops at the inferred default-seat-at-mint: read ledger heat → pick cooler lane → stamp default seat, reusing E-069/E-070 plumbing. The FULL demand signal is SEVERAL EPICS (PE-7): (a) THIS epic — overflow-aware default seat at mint from ledger-inferred heat; (b) a follow-on epic to CAPTURE the per-lane 429/usage/reset substrate in the ledger, if the prerequisite check shows it's missing; (c) a follow-on epic for LIVE mid-run rerouting when a 429 fires during an in-flight run (that reaches into the executor loop and borders lisa's job — deferred deliberately); (d) a follow-on to surface quota-per-reset-window as a first-class denomination in the budget gesture itself. Respects N2 — this is inference-at-mint written onto the board, NOT a live lane-heat dashboard or monitor UI; and N4 — Vend infers and writes the seat, lisa still dispatches and executes; Vend does not intercept 429s at runtime here. Out of scope: adding lanes beyond claude|codex; per-ticket (vs per-gesture) seat inference; changing how lisa dispatches; and any live/mid-run rerouting.
