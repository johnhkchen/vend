## Intent — the bigger-picture play

The durable fix for the **D-005 commit-discipline andon**. lisa advances a ticket's
`phase` on *artifact detection*, not on commit — so a session can mark a ticket "done"
while its code sits uncommitted in the working tree. It bit three times this session
(E-001, E-006, E-007 residuals), once leaving **`HEAD` non-building** (`cast.ts`
committed without its `play.ts`). The fix: a `bun run check:committed` script (the
shared definition — the Central Rule) that **fails when source is uncommitted or
untracked**, fired by a **lisa hook** at session stop so the divergence can't pass
silently.

## Value to the design

A gate that makes the andon impossible to ignore (**P3** — "done" carries an
enforceable contract). It's the exact thing the protocol learning told us to build
(`project-steering.md`: *"verify git, not `lisa status`"*) — automated, so the human
doesn't have to catch it by hand every loop (which is *itself* the supervision the
vision exists to remove).

## Done looks like

- `bun run check:committed` exits **non-zero** when `src/` / `baml_src/` / `ci/` have
  uncommitted or untracked changes; gitignored runtime (`.vend/*`, `node_modules`,
  `baml_client`) is never a failure.
- a **lisa hook** (`.lisa/hooks/on-stop.sh`) runs it at session stop and surfaces a
  clear uncommitted-code andon (the offending paths).
- a dirty-tree demo shows the andon fires; a clean tree passes.

## Context & constraints

- **Cites:** `D-005` / `D-010` (the recurring andon + the E-007 regression);
  `ci-strategy.md` (the Central Rule — logic lives in `check:*` scripts; here the hook
  only invokes); `playbooks/project-steering.md` → Protocol learnings ("done ≠
  committed"); `.lisa/hooks/` (the enforcement home); `.gitignore` (the allowed-runtime
  exclusions).
- **Design call (honest):** an *uncommitted-code* check is about the **host's git
  working-tree state**, which a **Dagger container cannot see** (it mounts a snapshot,
  always "clean" relative to itself). So — unlike E-002's `test` gate — its enforcement
  point is a **lisa hook**, not a Dagger sub-class. Don't force it into the container;
  that would be well-formed-wrong.
- **Non-goals:** don't flag gitignored runtime as uncommitted (N-confusion); not an
  executor (**N4**).
- **First slice:** the `check:committed` script + the lisa-hook trigger. **Out**
  (follow-ups): a container-side *HEAD-builds* gate (check out the committed HEAD and
  build, catching inconsistent partial commits like `cast.ts`-without-`play.ts`), and
  hard-blocking if the hook can only warn.
