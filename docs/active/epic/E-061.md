## Intent — the bigger-picture play

Make vend installable the way lisa already is: `brew install johnhkchen/vend/vend` lands a self-contained binary, and `vend init --template <name>` lays down a ready workspace — no repo clone, no Doppler, bring-your-own `claude login`. Today vend has no install path at all (no release CI, package.json still `private`/`0.0.0`/no `bin`), which blocks the kitchen dogfood and gates every future user. The play is to mirror johnhkchen/homebrew-lisa exactly — `bun build --compile` per-platform binary, tarball + sha256, a GitHub release, a `vend.rb` in its own tap — and to extend the existing E-058 `vend init --template` seam (NOT a new command) so a brew-installed vend can make a workspace. Because vend's binary bundles BAML (~108 MB native addon, vs lisa's 2.5 MB pure build), cross-compiling all four targets is risky; the MVP ships the cook/dev's own platform first (arm64-mac), others later.

## Value to the design

Realizes the distribution capability that turns vend from a clone-and-run dev artifact into a product someone can install and use locally. Advances P5 (local-first): a self-contained per-platform binary, offline-usable on one machine, owning its own state, with no cloud/Doppler dependency and the executor login supplied by the user. Reuses lisa's verified mechanism rather than inventing one, and reuses the existing init-template seam rather than growing surface — keeping the install path a maintained, repeatable pipeline (permanent), not a one-off.

## Done looks like

On a fresh arm64-mac with no vend checkout, `brew install johnhkchen/vend/vend` succeeds, `vend --version` reports a real semver (not 0.0.0), and `vend init --template <name>` lays down a workspace in an empty directory — all without cloning the repo and without Doppler, using only the user's own `claude login`. The release that produced the binary carries the platform tarball + a verified sha256, and the tap formula selects/verifies/installs it the way lisa's does.

## Context & constraints

Ground truth: lisa's mechanism is verified live (`johnhkchen/homebrew-lisa/Formula/lisa.rb`, lisa 0.3.0 — compiled binary, 4 targets, own tap, sha-per-variant, MIT, no livecheck). vend's gaps are concrete and pinned in pm/plan-kitchen-dogfood.md: no `.github/workflows` (release CI absent), `package.json` is `private:true` / version `0.0.0` / no `bin`, and the `vend init --template` seam exists from E-058 to extend. SCOPE: ship the install path + template-seam extension for ONE platform (the cook/dev's; need their OS/arch — likely arm64-mac), mirroring lisa's own-tap pattern. PREREQUISITES (named, not hidden scope): (1) the cook/dev's OS/arch input, to pick the first binary target; (2) the kitchen seed CONTENT is a SEPARATE epic (E-062) and a parallel prerequisite — this epic ships the seam and the install path and can validate against a minimal/placeholder template, it does not author the kitchen workspace content. OUT OF SCOPE: the full 4-platform binary matrix (deferred past MVP because BAML's native addon makes cross-compile non-trivial — vend binary ~108 MB), the menu-render drive slice (the downstream MVP first-drive), and a consolidated tap (lean is mirror-lisa two-tap, consolidate later). NON-GOALS: violates none — N4 holds (this is packaging; the user's own claude login does the executing, vend still only clears/allocates). RIGHT-SIZE (PE-7): one ~2-hour feature block IS achievable at single-platform scope; if the cook/dev's platform turns out to need cross-compile work, or the full 4-target matrix is pulled in, that is a second epic — do not inflate this card to cover it.
