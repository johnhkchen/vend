## Intent — the bigger-picture play

Today the typed DAG is branchless: DagEdge is {from,to} and runGraph fans a node's produced output to EVERY downstream — the graph can only express 'do all of these next', never 'do this OR that depending on what came back'. That is the named E-046 follow-on and the obvious next increment on the just-finished DAG substrate (E-046/E-047/E-048). Give each edge a predicate over the upstream node's produced result: only out-edges whose predicate holds thread downstream, and the branch-not-taken's subgraph is skipped — landing in `skipped` with an andon — by reusing the existing halt-dependent-subgraph machinery, so nothing is ever silently dropped. The same branching must hold in castGraph (the concurrent wave dispatcher), not just the sequential runner. This is the capability that turns the typed graph from a fixed pipeline into something that can encode real conditional process, which is what v1 graph-orchestration is for.

## Value to the design

Realizes conditional control flow on the orchestration substrate — the difference between a playbook that runs a fixed sequence and one that encodes branching judgment. It advances the core feature (richer, more faithful clearing of intent into graph-structured work) and: P1 — branch predicates are declared once at authoring, never re-specified per run; P3 — a not-taken branch is an observable `skipped`+andon outcome, not a silent gap, so 'this path didn't run' is told apart from 'this path was forgotten'; P6 — the predicate evaluation stays in the pure core with the impure shell untouched, keeping the engine executor-agnostic.

## Done looks like

A node can carry conditional out-edges, and at the epic level it is observably true that: a node's produced result selects which downstream edges fire; nodes on a not-taken branch land in `skipped` with an andon (never silently missing); the behavior is identical under both runGraph and castGraph (sequential and concurrent wave dispatch); the pure-core/impure-shell split is preserved; and the whole thing is proven by deterministic stub-node tests with no live model in the loop, green in the existing check gate.

## Context & constraints

Ground against src engine: `dag-core.ts` (DagEdge today is just {from,to}), `graph-core.ts` (runGraph fan-out to all downstream; castGraph wave dispatcher with the shared wallet from E-048), and the existing halt-dependent-subgraph / `skipped` + andon machinery this epic REUSES rather than reinvents — that reuse is the design spine, not new infrastructure. Prerequisites, all already satisfied: E-046/E-047/E-048 (the DAG substrate, join/fan-out/halt semantics, and cross-branch budget wallet) are done. Boundaries / out of scope: this is intent only, not decomposition (DecomposeEpic owns stories/tickets); no live-model execution — tests are deterministic stub nodes (respects the 'prove it without a live model' constraint); does not regress N4 — Vend still only clears and shapes the graph, lisa executes. Right-sizing (PE-7): the signal itself calls this 'about one or two feature blocks'; it sits at the larger end of a single epic — if mirroring into castGraph plus the skip/andon wiring proves to exceed one ~2-hour block at decompose time, it splits cleanly along the runGraph-vs-castGraph seam into two, but it is proposed as one epic.
