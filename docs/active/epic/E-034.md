## Intent — the bigger-picture play

E-033 **prevents** red commits going forward (the pre-commit hook blocks them). But prevention has two
gaps it cannot close: **history that predates the gate** still contains red commits (the literal
motivating case — `3dfb95f`, the T-031-01 commit that shipped a failing test, is still in `main`'s
history), and the hook can be **bypassed** (`git commit --no-verify`) or simply **absent** on a clone
that never ran `hooks:install`. The `check:hooks` guard nags about the last case, but nothing *audits
what already happened*.

This builds the backstop: **`check:history`** — resolve a bounded commit range, rebuild + test **each**
commit in an isolated git worktree (the E-010 `buildCommittedHead` machinery, generalized to an
arbitrary commit-ish), and **flag any commit that wasn't green**. Exit non-zero if the range contains a
red commit, so it can run in CI as the periodic audit.

**The load-bearing design call: a linear walk, not a bisect.** Finding *all* red commits is not what
`git bisect` does — bisect binary-searches for the *first* good→bad transition, assuming the property
is **monotonic** (once broken, stays broken). Red-ness is **non-monotonic**: a commit can be green, the
next red, the next green again (exactly the mid-ticket pattern E-033 addresses). So the correct audit is
a **linear sweep** that checks every commit in the range. (`git bisect run` remains the right tool for a
*different* question — "which commit first broke *this specific* test" — and is named as a downstream
convenience, not this slice.) The kaizen said "bisect" loosely; this corrects it to the honest tool.

## Value to the design

Completes the gate story: E-033 stops new red commits, E-034 finds the ones already there or slipped
through — P3 (consistency / an honestly-green history, `git bisect` stays meaningful because the audit
keeps it true) and P4 (loop integrity / the autonomous history is auditable, not assumed). Together they
make "every commit is green" a **checkable claim**, not a hope.

## Done looks like

- **A `check:history [<range>]` sweep** — resolves a bounded range (default + explicit `main..HEAD` /
  last-N / `--since`), rebuilds + tests each commit in an **isolated worktree** (no working-tree
  disturbance, cleaned up in all paths — the E-010 precedent), and reports per-commit green/red.
- **Honest bounding (no silent caps)** — the range is bounded (a runaway-history guard); whatever is
  **not** covered is stated loudly (the "covered N of M; bounded at K" note), never silently dropped.
- **A pure `history-core.ts`** — `classifyHistory(results)` → the report + the exit decision (any red ⇒
  non-zero) + honest-empty (empty range); unit-tested (the `committed-core.ts`/`head-build-core.ts`
  precedent).
- **Proven free + deterministic:** the sweep over a range containing the known-red `3dfb95f` **flags it
  red** (the audit finds the very escape that motivated E-033); a known-green range reports clean — no
  LLM cast. `bun run check:*` green.

## Context & constraints

- **Cites:** `src/ci/head-build-core.ts` + `src/ci/check-head.ts` (`buildCommittedHead` ~53 — the
  isolated-worktree build-a-commit machinery to **generalize** from `HEAD` to an arbitrary commit-ish;
  `classifyBuildOutcome`, the `*-core.test.ts` precedent); `src/ci/committed-core.ts` (the pure-core /
  thin-invoker split); `package.json` (the `check:*` script block — add `check:history`); E-033
  (`precommit-core.ts` — the per-commit gate this audits *retroactively*; run the same gate per commit);
  `docs/active/epic/E-010.md` (the worktree-build lineage); `docs/knowledge/ci-strategy.md` (hook /
  check / core conventions).
- **What to run per commit:** the E-033 contract is `check:test`, but a **fresh worktree needs
  generated code present** — run a command that includes codegen (the full `check`, or
  `baml:gen && check:test`) so the checkout is valid, exactly as `buildCommittedHead` runs the full
  `check` today. Note the per-commit cost (worktree + install + build ≈ a few seconds each) — this is a
  **periodic/CI audit, not per-commit** (that is E-033's job).
- **Bounded by default (no silent caps):** an unbounded sweep over deep history is a footgun; require an
  explicit range or apply a default bound (e.g. last N), and **log what was and wasn't covered**. An
  audit that silently checked 20 of 200 commits would read as "all green" dishonestly.
- **Non-goals:** a true `git bisect run` integration (the monotonic "when did X first break" question —
  a different tool, downstream); auto-*fixing* or rewriting history (report only — rewriting shared
  history is N-territory); running per-commit in the hot path (E-033 owns prevention; this is periodic);
  parallelizing the sweep (sequential first slice, the `buildCommittedHead` precedent — speed is a
  later pull). Not an executor (**N4**).
- **First slice = the pure `history-core` + the `check:history` bounded linear sweep reusing the E-010
  worktree machinery + the honest-bound note.** **Out:** bisect integration, parallelism, auto-fix.
